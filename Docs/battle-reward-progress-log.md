- 단계마다 변경 파일, 자동 테스트, 실제 화면과 패키지·씬·외부 에셋 변경 여부를 기록한다.

- AI 입력 원문은 그대로 복사하지 않고 목적, 지시, 산출물과 사람의 결정을 정제해 기록한다.
- 기획·코드·검증의 최종 승인 책임자는 이천서로 기록한다.
- 사용자의 명시적 요청 전에는 스테이징·커밋·푸시하지 않는다.

RW-00 착수 당시 작업 트리에는 이전 카드 사용 검증 문서·테스트와 사용자가 편집한 기획 문서 및 DOCX 파일이 함께 존재한다. 이번 작업은 기존 변경을 되돌리거나 DOCX 파일을 수정하지 않고 전투 보상 Markdown 문서와 연결 기록만 추가·갱신한다.

## 5. 단계별 기록

### RW-00 — 구현 기준 확정

#### 수행 내용

- `rule.md`와 전체 게임 기획서에 최종 보스도 보상을 제공한다고 명시했다.
- 보스 승리는 즉시 런 승리가 아니라 높은 등급 카드 3장 중 선택 또는 건너뛰기 뒤 `RunVictory`로 전환하도록 정했다.
- 스테이지 진행 문서에 완료된 SP-04 코드와 후속 RW 작업의 책임 경계를 명시했다.
- 전투 보상 기획서, 개발 명세서, 구현 계획서와 진행 기록을 작성했다.
- 일반·높은 등급 풀, 보상 후보·제안, 덱 추가·초기화, 상태·세션·UI API와 테스트 기준을 작성했다.
- README, AI 활용 기술서와 이천서 역할 기술서의 연결 기록을 갱신 대상으로 확정했다.

#### 변경 범위

- Markdown 기획·개발 문서만 변경
- 게임 코드 변경 없음
- 테스트 코드 변경 없음
- Unity 씬·프리팹 변경 없음
- 패키지 변경 없음
- 외부 에셋 추가 없음
- 기존 DOCX 파일 변경 없음

#### 검증

- `rule.md`, 전체 게임 기획서와 스테이지 진행 문서의 보상 순서 대조
- `StageProgressionState`, `RunProgress`, `PlayerRunState`, `StageProgressionSession`, 카드 카탈로그와 UI 구조 정적 대조
- 전투 보상 문서 4종의 작업 ID, 상태 전이, 포함·제외 범위와 테스트 단계 대조
- Git diff 공백 오류와 문서 링크 확인
- 문서 전용 단계이므로 Unity Editor와 EditMode 테스트는 다시 실행하지 않음

#### 남은 위험

- 정식 카드 희귀도 데이터가 생기면 프로토타입 높은 등급 목록을 이관해야 한다.
- 엘리트 상대 선택 구조가 아직 없어 RW-03에서는 명시적 보상 등급 경계까지만 연결할 수 있다.
- 최종 보스 보상 카드는 현재 추가 전투에 쓰이지 않으므로 결과 화면·메타 성장 기획에서 의미를 보강할 수 있다.
- 휴식·상점·사건의 순서와 보상 이후 이동은 별도 런 구조 명세가 필요하다.

### RW-02 — 보상 진행 상태와 처리

#### 수행 내용

- `RewardSelection`을 추가해 보상 해결 전 다음 진행·패배·재시작·새 보상 입력을 잠갔다.
- 제안과 완료 목적지를 보존하는 `PendingBattleReward`, 처리 결과를 남기는 `BattleRewardResolution`을 추가했다.
- `RunProgress`에 보상 시작·선택·건너뛰기 API를 추가했다.
- 일반전은 `StageCleared`, 최종 보스는 높은 등급 제안 처리 뒤 `RunVictory`로만 이동하게 했다.
- 잘못된 목적지·옵션과 중복 입력은 덱·상태·최근 결과를 바꾸지 않게 했다.
- 재시작에서 보류 제안과 최근 해결 결과를 초기화했다.
- 기존 상태·표시 테스트를 새 공개 보상 API 기준으로 마이그레이션했다.

#### 변경 파일

- 신규 런타임: `BattleRewardCompletionTarget.cs`, `PendingBattleReward.cs`, `BattleRewardResolution.cs`
- 수정 런타임: `RunProgress.cs`, `StageProgressionState.cs`, `StageProgressionSession.cs`
- 표시 호환: `StageProgressionPresentation.cs`의 보상 선택 메시지
- 신규 테스트: `BattleRewardStateTests.cs`
- 수정 테스트: `StageProgressionStateTests.cs`, `StageProgressionPresentationTests.cs`
- 신규 C# 파일의 Unity `.meta`, 전투 보상 문서 4종과 공통 기록 문서

#### 검증

- RW02-U01~RW02-U07 신규 테스트 7/7 통과
- StageProgression EditMode 49/49 통과
- 전체 EditMode 171/171 통과, 실패·건너뜀 0
- Unity 6000.3.10f1 로컬 MCP로 스크립트 갱신·컴파일과 테스트 실행
- Console 컴파일 오류 0
- 씬·프리팹·패키지·외부 에셋 변경 없음

#### 범위 경계와 남은 위험

- `StageProgressionSession`은 RW-03 전까지 내부 `TryCompleteCurrentStageWithoutReward()` 호환 경로를 사용한다. 따라서 현재 실제 씬의 승리는 아직 보상 제안을 생성하지 않는다.
- 엘리트 스테이지 타입이 아직 없어 높은 등급 연결은 명시적 제안 등급 경계로만 검증했다.
- 보상 후보 표시·선택 버튼·건너뛰기 화면은 RW-04 범위다.

#### 추천 커밋 제목

`승리 결과를 보상 선택이 끝난 뒤에만 확정하도록 진행 상태를 연결`

### RW-03 — 전투 결과와 세션 통합

#### 수행 내용

- 전투 승리 시 현재 영혼을 먼저 `PlayerRunState`에 동기화하고 보상 제안을 생성했다.
- 세션별 보상 생성기 하나를 유지해 같은 승리에서 제안이 중복 생성되지 않게 했다.
- 일반 스테이지는 기본 일반 등급, 최종 보스는 강제 높은 등급과 런 승리 목적지를 사용하게 했다.
- 미래 엘리트는 스테이지 타입을 늘리지 않고 보상 등급 결정 함수를 세션에 주입할 수 있게 했다.
- 선택·건너뛰기 세션 API를 추가하고, 보상 처리 전 다음 전투 생성을 차단했다.
- 선택한 카드는 다음 전투 생성 시 현재 런 덱에 포함되고 건너뛰기는 덱 장수를 유지하게 했다.
- 런 재시작은 최초 덱 복구 뒤 새 전투를 생성하게 했다.
- RW-02의 내부 무보상 완료 경계를 제거하고 기존 세션 회귀를 새 흐름으로 이전했다.
