## 1. 기록 원칙

- 계획과 실제 구현을 구분한다.
- 코드가 없는 문서 단계는 구현 완료로 기록하지 않는다.
- 단계마다 변경 파일, 자동 테스트, 실제 화면과 패키지·씬·외부 에셋 변경 여부를 기록한다.
- 실패한 검증과 미해결 위험을 숨기지 않는다.
- AI 입력 원문은 그대로 복사하지 않고 목적, 지시, 산출물과 사람의 결정을 정제해 기록한다.
- 기획·코드·검증의 최종 승인 책임자는 이천서로 기록한다.
- 사용자의 명시적 요청 전에는 스테이징·커밋·푸시하지 않는다.

## 3. 착수 기준선

### 3.1 완료된 선행 작업

- 코어 루프 1~4단계 완료
- 런·스테이지 진행 SP-00~SP-04 완료
- 전투 행동 BA-00~BA-05 완료
- 카드 사용 CU-00~CU-06 완료
- CU-06 기준 신규 반복 회귀 5/5, CoreLoop 122/122, StageProgression 34/34, 전체 EditMode 156/156 통과
- `CoreLoopTest`, `StageTest` 씬 문제 0과 Console Error/Warning 0 확인

### 3.2 RW-00 시점 코드 상태

- 전투 승리는 보상 없이 즉시 `StageCleared` 또는 `RunVictory`가 된다.
- `StageProgressionState`에 보상 선택 상태가 없다.
- `PlayerRunState.Deck`은 읽기 전용이며 런 중 추가할 수 없다.
- 런 재시작은 영혼만 복구한다.
- 카드 정의 키와 카드 효과 카탈로그는 이미 존재한다.
- `StageTest` 진행 화면과 세션 연결은 존재한다.
- 엘리트 여부를 나타내는 스테이지 종류나 상대 선택 결과는 아직 없다.

### 3.3 Git 주의사항

RW-00 착수 당시 작업 트리에는 이전 카드 사용 검증 문서·테스트와 사용자가 편집한 기획 문서 및 DOCX 파일이 함께 존재한다. 이번 작업은 기존 변경을 되돌리거나 DOCX 파일을 수정하지 않고 전투 보상 Markdown 문서와 연결 기록만 추가·갱신한다.

## 5. 단계별 기록

### RW-00 — 구현 기준 확정

#### 수행 내용

- `rule.md`와 전체 게임 기획서에 최종 보스도 보상을 제공한다고 명시했다.
- 보스 승리는 즉시 런 승리가 아니라 높은 등급 카드 3장 중 선택 또는 건너뛰기 뒤 `RunVictory`로 전환하도록 정했다.
- 스테이지 진행 문서에 완료된 SP-04 코드와 후속 RW 작업의 책임 경계를 명시했다.
- 전투 보상 기획서, 개발 명세서, 구현 계획서와 진행 기록을 작성했다.
- 일반·높은 등급 풀, 보상 후보·제안, 덱 추가·초기화, 상태·세션·UI API와 테스트 기준을 작성했다.
- README, AI 활용 기술서와 이천서 역할 기술서의 연결 기록을 갱신 대상으로 확정했다.

#### 변경 범위

- Markdown 기획·개발 문서만 변경
- 게임 코드 변경 없음
- 테스트 코드 변경 없음
- Unity 씬·프리팹 변경 없음
- 패키지 변경 없음
- 외부 에셋 추가 없음
- 기존 DOCX 파일 변경 없음

#### 검증

- `rule.md`, 전체 게임 기획서와 스테이지 진행 문서의 보상 순서 대조
- `StageProgressionState`, `RunProgress`, `PlayerRunState`, `StageProgressionSession`, 카드 카탈로그와 UI 구조 정적 대조
- 전투 보상 문서 4종의 작업 ID, 상태 전이, 포함·제외 범위와 테스트 단계 대조
- Git diff 공백 오류와 문서 링크 확인
- 문서 전용 단계이므로 Unity Editor와 EditMode 테스트는 다시 실행하지 않음

#### 남은 위험

- 정식 카드 희귀도 데이터가 생기면 프로토타입 높은 등급 목록을 이관해야 한다.
- 엘리트 상대 선택 구조가 아직 없어 RW-03에서는 명시적 보상 등급 경계까지만 연결할 수 있다.
- 최종 보스 보상 카드는 현재 추가 전투에 쓰이지 않으므로 결과 화면·메타 성장 기획에서 의미를 보강할 수 있다.
- 휴식·상점·사건의 순서와 보상 이후 이동은 별도 런 구조 명세가 필요하다.

#### 추천 커밋 제목

`보스까지 이어지는 전투 보상 흐름을 구현 기준으로 확정`

### RW-01 — 보상 데이터·생성·런 덱 기반

#### 수행 내용

- 일반 10개와 높은 등급 6개 정의 키를 명시적으로 검증·노출하는 보상 카탈로그를 추가했다.
- 옵션 ID 0~2와 중복되지 않는 카드 정의 3개를 가진 불변 보상 제안을 추가했다.
- 같은 시드와 요청 순서에서 같은 후보 순서를 생성하고 제안 ID를 증가시키는 생성기를 추가했다.
- `PlayerRunState`의 최초 덱 스냅샷과 현재 런 덱을 분리했다.
- 보상 카드에 현재 런 최대 카드 ID 다음 값을 발급하고 같은 정의의 중복 획득을 허용했다.
- 재시작 시 최대 영혼, 최초 덱과 다음 카드 ID를 함께 복구하게 했다.
- 내부 덱 변경 API는 StageProgression 테스트 어셈블리에만 직접 공개했다.

#### 변경 파일

- 런타임: `BattleRewardTier.cs`, `BattleRewardCatalog.cs`, `BattleRewardOption.cs`, `BattleRewardOffer.cs`, `BattleRewardGenerator.cs`
- 수정: `PlayerRunState.cs`, `AssemblyInfo.cs`
- 테스트: `BattleRewardFoundationTests.cs`
- 각 신규 C# 파일의 Unity `.meta`
- 전투 보상 문서 4종, README, AI 활용·팀 역할·프로젝트 구조 기록

#### 검증

- RW01-U01~RW01-U08 신규 테스트 8/8 통과
- StageProgression EditMode 42/42 통과
- 전체 EditMode 164/164 통과, 실패·건너뜀 0
- Unity 컴파일 오류 0, 테스트 실행 종료 코드 0
- Unity 배치 로그에서 라이선스 토큰 갱신 메시지를 확인했으나 컴파일과 테스트 결과에는 영향 없음
- 씬·프리팹·패키지·외부 에셋 변경 없음

#### 범위 경계

- `RewardSelection` 상태와 보류 보상은 아직 추가하지 않았다.
- 실제 승리 후 보상 생성, 선택·건너뛰기와 `StageCleared`·`RunVictory` 전환은 아직 연결하지 않았다.
- 화면 표시와 입력은 변경하지 않았다.

#### 추천 커밋 제목

`전투 보상을 재현 가능하게 생성하고 런 덱에 안전하게 담을 기반을 마련`

## 7. 다음 단계 착수 조건

RW-02 착수 시 다음을 먼저 확인한다.

1. RW-01 신규 8개와 전체 164개 테스트 기준선을 유지한다.
2. 기존 `TryCompleteCurrentStage()`의 즉시 완료 테스트와 호출부를 새 상태 전이에 맞춰 함께 변경한다.
3. `RewardSelection`, 완료 목적지와 보류 제안 테스트를 구현보다 먼저 작성한다.
4. 잘못된 옵션·중복 선택·보상 중 진행 입력이 덱과 상태를 바꾸지 않는지 검증한다.
5. 세션·UI 연결은 RW-03·RW-04로 남긴다.

RW-02의 첫 구현 대상은 `RunProgress`의 보상 시작·선택·건너뛰기와 일반·보스 완료 목적지다. 실제 전투 결과에서 제안을 생성하는 세션 연결은 RW-03으로 넘긴다.
