> 최종 갱신: 2026-07-20

## 1. 계획 목적

카드 사용 시스템을 데이터 기반, 효과 처리, 카드별 구현, 화면·런 연결과 전체 검증으로 나눠 구현한다. 각 단계는 문서 확인 → 테스트 기준 작성 → 최소 구현 → Unity 검증 → 기록 → 커밋 순서로 독립 완료한다.

한 단계의 완료 증거가 없으면 다음 단계 코드를 미리 구현하지 않는다. 단계별 커밋을 권장하며, 사용자가 별도로 요청하지 않는 한 AI는 스테이징하거나 커밋하지 않는다.

## 2. 실행 방식

각 단계에서 다음 절차를 반복한다.

1. `card-use-design.md`와 개발 명세의 해당 범위를 확인한다.
2. 직전 단계의 Git 상태와 Unity 회귀 기준을 확인한다.
3. 실패·경계 조건을 포함한 EditMode 테스트를 먼저 고정한다.
4. 해당 단계에 필요한 최소 코드만 구현한다.
5. 스크립트 진단, 대상 테스트와 전체 EditMode 테스트를 실행한다.
6. UI가 포함된 단계는 실제 Game View와 씬·Console을 확인한다.
7. 진행 기록, AI 활용 기술서와 이천서의 실제 담당 기록을 갱신한다.
8. 변경 범위에 맞는 한글 Lore 커밋 제목과 검증 본문을 추천한다.

## 5. CU-01 — 카드 정의와 사용 상태 기반

### 목표

숫자만 가진 기존 카드를 안정된 정의와 전투 중 사용 상태를 가진 구조로 확장한다. 완성 카드 행동이나 UI는 아직 추가하지 않는다.

### 작업

- `CardDefinition`, 발동·효과·사용 상태 열거형 추가
- 기본 숫자 카드 카탈로그 추가
- `BlackjackCard`에 정의와 사용 상태 연결
- 기존 생성자 호환 유지
- `RunCardDefinition`에 정의 키 보존
- `StageBattleFactory`의 키 해석 연결
- 손에 들어올 때 사용 상태 초기화
- 정의·호환·재드로우·런 변환 테스트 추가

### 제외

- 카드 사용 공개 API
- 효과 선택 상태
- 카드별 효과
- 화면 버튼

### 완료 게이트

- 기존 숫자 카드 테스트가 의미 변경 없이 통과한다.
- 기본 카드 열 장의 정의가 표와 일치한다.
- 수동 카드만 사용 가능 상태로 초기화된다.
- 사용 완료 상태가 같은 손에서 유지되고 재드로우 시 초기화된다.
- 런 카드 정의 키가 전투까지 보존된다.
- 대상·전체 EditMode 테스트가 통과한다.

### 예상 변경 경로

- `Assets/01. Scripts/Runtime/CoreLoop/`
- `Assets/01. Scripts/Runtime/StageProgression/RunCardDefinition.cs`
- `Assets/01. Scripts/Runtime/StageProgression/StageBattleFactory.cs`
- `Assets/Tests/EditMode/CoreLoop/`
- `Assets/Tests/EditMode/StageProgression/`
- 관련 문서

### 완료 결과

- 숫자 1~10을 안정된 정의 키·표시 이름·발동 유형·효과 유형에 연결했다.
- 기존 숫자 생성자를 유지하고 `BlackjackCard`에 정의와 사용 상태를 추가했다.
- 손 진입 경계에서 수동 카드만 `Available`로 초기화하며, `Available → Resolving → Used` 전이와 같은 손에서의 재사용 차단을 구현했다.
- 런 덱은 `DefinitionKey`를 보존하고 전투 생성 시 카탈로그로 해석한다. 알 수 없는 키는 예외로 실패한다.
- 신규 19개 테스트를 포함한 전체 EditMode 101/101과 Unity Console Error/Warning 0을 확인했다.
- 카드 효과, 공개 사용 API, 화면·씬·패키지·외부 에셋은 변경하지 않았다.

### 추천 커밋 제목

`카드 효과를 안전하게 확장할 수 있도록 정의와 사용 상태를 분리`

## 6. CU-02 — 효과 선택과 처리 기반

### 목표

카드 사용 승인, 선택 대기, 명령 적용, 즉시 종료와 차례 복귀를 카드별 콘텐츠 없이 검증 가능한 기반으로 만든다.

### 작업

- `PlayerResolvingCardEffect` 상태 추가
- 사용 가능 판정과 불가 사유 모델 추가
- `PendingCardEffect`·선택 옵션 모델 추가
- `TryBeginPlayerCardUse`, `TryResolvePlayerCardChoice` 추가
- 손패 ID 조회·인출과 덱 임시 분리·반환 기능 추가
- 공통 효과 명령 경계와 카드 효과 결과 모델 추가
- 직접 카드 효과 버스트 원인 기록
- 선택 중 다른 행동 차단
- `CoreLoopSession` 전달 추가

### 제외

- 최종 카드 4종의 완전한 규칙
- View 버튼과 런 전달
- 자동 발동·계약

필요하면 테스트 전용 단순 효과 정의를 사용하되 출시 카탈로그에 가짜 카드를 남기지 않는다.

### 완료 게이트

- 잘못된 요청이 상태를 바꾸지 않는다.
- 승인된 비공개 카드만 공개된다.
- 선택 중 다른 행동과 중복 카드 사용이 차단된다.
- 유효 선택 뒤 효과가 한 번만 계속된다.
- 효과 종료 시 적 차례가 실행되지 않는다.
- 카드가 어느 영역에도 중복 소유되지 않는다.
- 대상·전체 EditMode 테스트가 통과한다.

### 완료 결과

- 카드별 콘텐츠와 분리된 사용 가능 판정·불가 사유 모델을 추가했다.
- 승인 전 검증은 읽기 전용으로 유지하고, 승인된 카드만 공개·`Resolving` 상태로 전이한다.
- 선택 없음, 단일 선택과 연속 선택을 같은 `CardEffectStep` 경계로 처리한다.
- 선택 대기 중 모든 다른 전투 입력을 잠그며 잘못된 옵션은 상태를 바꾸지 않는다.
- 효과 결과와 `CardEffectBust` 종료 원인을 기록하고, 라운드 종료 시 적 차례를 건너뛴다.
- 손패 ID 조회·인출과 덱 상단 카드 임시 분리·순서 보존 반환을 원자적으로 제공한다.
- 출시 카탈로그에는 가짜 효과를 추가하지 않고 테스트 전용 내부 처리기 주입으로 기반만 검증했다.
- CU-02 신규 16개, CoreLoop 87/87, 전체 EditMode 117/117과 Console Error/Warning 0을 확인했다.
- 실제 카드 효과, View·진행 연결, 씬·패키지·외부 에셋은 변경하지 않았다.

### 추천 커밋 제목

`카드 효과를 안전하게 처리하도록 선택과 완료 경계를 고정`

## 7. CU-03 — 리볼버 세로 기능

### 목표

리볼버를 첫 실제 카드로 구현해 정의 → 사용 → 숫자 선택 → 효과 판정 → 라운드 결과 → 세션 전달의 전체 규칙 경로를 검증한다.

### 작업

- 리볼버 사용 조건과 숫자 선택 1~10
- 상대 비공개 숫자의 규칙 계층 내부 비교
- 성공 시 카드 효과 버스트와 적 영혼 피해
- 실패 결과와 적 차례 진행
- 사용 완료와 재사용 차단
- 카드 효과 결과의 안전한 표시 데이터 준비
- CoreLoop 세션 수준 테스트

### 제외

- 최종 View 입력
- 런 진행 전달
- 수정 구슬·해머·나이프

### 완료 게이트

- 성공·실패 경로가 고정 덱에서 재현된다.
- 성공 시 적 영혼 1 피해와 전투 승리가 정확하다.
- 실패 시 영혼 변화 없이 적 차례가 한 번 실행된다.
- 상대 비공개 숫자가 ViewModel·로그·테스트 실패 문구에 노출되지 않는다.
- 사용 완료 카드를 다시 사용할 수 없다.
- 대상·전체 EditMode 테스트가 통과한다.

### 완료 결과

- 기본 효과 처리기 목록에 숫자 7·8이 공유하는 `AutoPistolEffectHandler`를 등록했다.
- 정상 라운드의 상대 비공개 카드 한 장을 대상으로 1~10만 선택지로 제공한다.
- 다중 비공개 카드 지원은 추가하지 않고, 대상이 없으면 승인 전 원자적으로 거절한다.
- 선언 숫자 비교는 규칙 계층 내부에서만 수행하며 비공개 카드를 공개하지 않는다.
- 성공 시 적 영혼 1 피해와 카드 효과 버스트 원인을 기록하고 적 차례 없이 다음 라운드·승리로 진행한다.
- 실패 시 피해 없이 카드를 사용 완료하고 적 차례를 정확히 한 번 실행한다.
- 공개 효과 결과에는 상대 비공개 숫자를 담지 않으며 같은 손의 사용 완료 카드는 재사용할 수 없다.
- CU-03 신규 8개, CoreLoop 95/95, 전체 EditMode 125/125와 Console Error/Warning 0을 확인했다.
- View·진행 세션·씬·패키지·외부 에셋은 변경하지 않았다.

### 추천 커밋 제목

`비공개 정보 노출 없이 리볼버의 추측 전투를 완성`

## 8. CU-04 — 나머지 일반 수동 카드

### 목표

수정 구슬, 위협용 해머와 보위 나이프를 구현해 덱 탐색, 카드 비용, 상태 취소·교체와 강제 드로우 명령을 완성한다.

### 작업

- 수정 구슬 2장 확인·0~1장 선택·덱 위 순서 복원
- 선택 카드 공개 추가와 플레이어 버스트 확인
- 해머의 자기 공개 카드 비용 선택
- 상대 스탠드 취소·비공개 카드 원자적 교체
- 나이프 공개 합 조건·강제 공개 드로우
- 강제 드로우 유지·버림 정책 경계와 기본 유지 정책
- 각 명령 뒤 즉시 버스트 확인
- 카드별 결정적 테스트

### 완료 게이트

- 수정 구슬의 모든 선택에서 카드 총수와 다음 드로우 순서가 보존된다.
- 해머는 자기 공개 카드만 선택하고 비용·추가 효과를 순서대로 처리한다.
- 나이프는 조건 외 사용을 거절하며 강제 드로우 결과를 정확히 처리한다.
- 효과 중 버스트 이후 후속 명령과 적 차례가 없다.
- 모든 카드가 사용 후 위치·합계·사용 상태 규칙을 지킨다.
- 대상·전체 EditMode 테스트가 통과한다.

### 완료 결과

- 수정 구슬의 임시 카드 소유권, 0장·첫째·둘째 선택과 다음 드로우 순서 복원을 구현했다.
- 해머 비용에 사용 카드 자체를 포함하고, 상대 스탠드 시 비공개 카드 정확히 한 장만 공개 없이 원자적으로 교체하도록 고정했다.
- 나이프의 공개 합계·덱 잔량 조건, 공개 강제 드로우와 교체 가능한 유지 정책 경계를 구현했다. 기본 정책은 항상 유지한다.
- 세 효과에서 발생한 숫자 합계 초과는 `NumericBust`로 즉시 종료하고 후속 적 차례를 실행하지 않는다.
- CU-04 전용 18/18, CoreLoop 113/113, 전체 EditMode 143/143과 최종 Console Error/Warning 0을 확인했다.
- 모델·CoreLoop 세션 단계만 완료했으며 실제 화면 입력과 런 진행 전달은 CU-05에 남겼다.

### 추천 커밋 제목

`일반 카드 효과가 같은 선택과 카드 이동 규칙을 따르도록 완성`

## 9. CU-05 — 화면과 런 전투 연결

### 목표

완성된 카드 규칙을 실제 전투 화면과 런 전투에서 사용할 수 있게 연결한다.

### 작업

- 플레이어 카드별 표시 모델과 사용 가능·완료·불가 사유
- 카드 사용 요청·효과 선택 View 이벤트
- 선택 중 전용 UI와 일반 행동 잠금
- 최근 카드 효과 결과 표시
- Controller의 독립·런 세션 전달
- `StageProgressionSession` 카드 사용·선택 전달과 종료 동기화
- Presenter·Controller·진행 통합 테스트
- `CoreLoopTest`, `StageTest` 실제 입력 검증

### 완료 게이트

- 화면에서 네 종류 카드를 선택하고 필요한 후속 선택을 완료할 수 있다.
- 사용 완료 카드와 조건 불충족 카드가 정확히 비활성화된다.
- 선택 중 히트·스탠드·폴드·체인지·다른 카드 입력이 실행되지 않는다.
- 독립 전투와 런 전투 결과가 일치한다.
- 카드 효과 전투 승리·패배가 런 진행과 영혼에 한 번만 반영된다.
- 대상·전체 EditMode, Game View, 양쪽 씬과 Console 검증이 통과한다.

### 완료 결과

- 카드별 표시 모델에 카드 ID·숫자·이름·사용 상태·사용 가능 여부와 불가 사유를 추가했다.
- 카드 사용·후속 선택 이벤트, 선택 전용 화면, 일반 행동 잠금과 안전한 최근 효과 결과 표시를 연결했다.
- Controller가 독립 전투와 런 전투 모두에서 카드 ID·선택 ID를 올바른 세션 API에 전달한다.
- `StageProgressionSession`이 승인된 카드 입력 뒤 기존 종료 동기화를 재사용해 승리·패배와 지속 영혼을 한 번만 반영한다.
- GUI 이벤트 중 표시 모델 교체에 대비해 선택 목록을 스냅샷으로 고정하고, 플레이 모드 입력 잠금은 다음 프레임에 해제한다.
- 신규 8개, 관련 Presenter·진행 통합 28/28, CoreLoop 117/117, StageProgression 34/34와 전체 EditMode 151/151이 통과했다.
- `StageTest`→`CoreLoopTest` 런 진입과 Game View의 수정 구슬·해머·리볼버·보위 나이프 사용·선택·최근 결과를 확인했고 양쪽 씬 문제 0·최종 Console Error/Warning 0을 확인했다.
- 씬 직렬화·패키지·외부 에셋 변경 없이 기존 즉시 모드 UI 구조를 재사용했다.

### 추천 커밋 제목

`카드 사용 선택이 실제 전투 화면과 런 결과까지 이어지도록 연결`

## 10. CU-06 — 전체 검증과 문서 마감

### 목표

카드 사용 시스템 전체를 반복 검증하고 완료 증거와 실제 담당 기록을 마감한다.

### 작업

- 카드 4종 정상·거절·버스트 경로 전체 회귀
- 공개·비공개 카드와 사용 상태 재드로우 반복
- 독립 전투 재시작과 런 승리·패배·재시작
- 고정 시드 반복에서 카드 총수·소유권 확인
- 숨은 정보가 화면·로그에 노출되지 않는지 확인
- 두 씬 검증과 Console 정리
- 계획·진행·AI 활용·이천서 역할 문서 최종 갱신

### 완료 게이트

- 신규·기존 EditMode 테스트 전부 통과
- 테스트 씬 문제 0
- 게임 관련 Console Error/Warning 0
- 실제 카드 4종 흐름 통과
- 런 승리·패배·재시작 통과
- 반복 검증에서 카드 중복·유실·입력 고착 0건
- 남은 제한과 후속 작업이 문서에 명시됨

### 완료 결과

- 독립 재시작, 수정 구슬 카드 소유권, 비공개 숫자 1~10 은닉, 런 승리·패배 재시작을 각각 10회 반복하는 테스트 5개를 추가했다.
- 신규 5/5, CoreLoop 122/122, StageProgression 34/34, 전체 EditMode 156/156이 통과했다.
- 실제 Controller·Runtime 흐름에서 최종 승리와 버스트 패배 뒤 `StageTest` 결과 화면·재시작·`CoreLoopTest` 새 전투 생성을 확인했다.
- 양쪽 씬 문제 0, Console Error/Warning 0이며 런타임·씬·패키지·외부 에셋 변경은 없다.
- 후속 범위는 자동 발동 카드, 계약, 카드 보상·강화·삭제, 최종 UI이며 이 계획에 이어서 구현된 것으로 간주하지 않는다.

### 추천 커밋 제목

`카드 사용의 규칙부터 런 결과까지 반복 검증해 작업을 마감`

## 11. 의존성

```text
CU-00
  └─ CU-01 카드 정의·사용 상태
       └─ CU-02 선택·효과 처리 기반
            └─ CU-03 리볼버 세로 기능
                 └─ CU-04 나머지 일반 카드
                      └─ CU-05 화면·런 연결
                           └─ CU-06 전체 검증
```

단계는 순차 진행한다. CU-01과 CU-02를 합치면 회귀 원인과 데이터 마이그레이션 문제가 섞이므로 분리한다.
