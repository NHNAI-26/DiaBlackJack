## 1. 계획 목적

전투 보상을 문서, 순수 데이터, 진행 상태, 세션 연동, 화면과 전체 검증 순서로 나누어 구현한다. 각 단계는 테스트와 기록까지 완료한 뒤 별도 커밋하는 것을 원칙으로 한다.

완료된 코어 루프, SP-00~SP-04, BA-00~BA-05와 CU-00~CU-06을 재작성하지 않는다. 필요한 변경은 전투 승리와 다음 진행 사이에 보상 상태를 삽입하고, 현재 런 덱에 선택 카드를 추가할 수 있게 하는 최소 경계로 제한한다.

## 2. 단계 운영 원칙

1. 단계 시작 전에 기획서와 개발 명세의 해당 범위를 확인한다.
2. 상태·실패 조건 테스트를 먼저 고정한다.
3. 단계 목표를 만족하는 최소 구현만 작성한다.
4. 관련 테스트, 전체 EditMode와 정적 검사를 순서대로 실행한다.
5. UI 단계는 실제 `StageTest`와 Console까지 확인한다.
6. 진행 기록, AI 활용 기술서와 이천서 역할 기술서를 갱신한다.
7. 사용자가 요청한 경우에만 스테이징·커밋·푸시한다.

## 4. RW-00 — 구현 기준 확정

### 목표

보스 승리를 포함한 모든 전투 보상의 순서, 후보 풀, 덱 추가, 상태 전이와 구현 단계를 확정한다.

### 완료 내용

- 일반·엘리트·최종 보스 승리 후 카드 3장 중 1장 선택 또는 건너뛰기를 명시했다.
- 일반·엘리트는 보상 뒤 `StageCleared`, 보스는 보상 뒤 `RunVictory`로 정했다.
- 일반 풀 10개, 높은 등급 풀 6개를 카드 정의 키로 임시 확정했다.
- 후보 정의 중복은 금지하고 기존 덱과 같은 정의 획득은 허용했다.
- 보상 선택 시 새 물리 카드 ID를 발급하고 새 런에서 최초 덱으로 복구하도록 정했다.
- 제거·강화·회복·유물은 사건·상점·휴식 작업으로 분리했다.
- RW-01~RW-05 구현·검증 게이트를 작성했다.

### 검증

- 기준 문서 간 보스 보상 순서와 범위 정적 대조
- 현재 `StageProgression`, 카드 정의와 UI 구조 대조
- Markdown 링크·작업 ID·담당자 표기 확인
- 문서 전용 단계이므로 Unity 테스트 미실행

### 추천 커밋 제목

## 5. RW-01 — 보상 데이터·생성·런 덱 기반

### 목표

진행 상태와 UI에 의존하지 않는 보상 데이터와 결정적 후보 생성, 현재 런 덱 추가·재시작 복구 기반을 만든다.

### 작업

1. `BattleRewardTier`와 기본 `BattleRewardCatalog`를 추가한다.
2. `BattleRewardOption`, `BattleRewardOffer` 불변 모델을 만든다.
3. 시드 또는 난수 공급자를 주입받는 `BattleRewardGenerator`를 구현한다.
4. `PlayerRunState`에 최초 덱 스냅샷과 현재 런 덱을 분리한다.
5. 보상 카드 추가와 단조 증가 ID 발급을 구현한다.
6. 재시작 시 영혼·덱·다음 ID가 최초 상태로 복구되게 한다.
7. RW01-U01~RW01-U08 테스트를 작성한다.

### 구현 결과

- 일반 10개·높은 등급 6개 정의 키를 검증하는 `BattleRewardCatalog`를 구현했다.
- 옵션 ID 0~2와 서로 다른 정의 키 3개를 보존하는 불변 제안 모델을 구현했다.
- 시드와 요청 순서가 같으면 같은 결과를 만드는 부분 Fisher-Yates 생성을 구현했다.
- `PlayerRunState`가 최초 덱과 현재 덱을 분리하고 보상 카드에 현재 최대 ID 다음 값을 발급하게 했다.
- 새 런에서 영혼, 최초 덱과 다음 카드 ID가 함께 복구되게 했다.
- 진행 상태·세션·화면은 변경하지 않았다.

### 검증 결과

- RW01-U01~RW01-U08: 8/8
- StageProgression EditMode: 42/42
- 전체 EditMode: 164/164
- Unity 컴파일 오류와 테스트 실패·건너뜀: 0
- 씬·프리팹·패키지·외부 에셋 변경: 없음

### 완료 게이트

- 두 풀 모두 정확히 명시된 유효 키만 가진다.
- 후보는 항상 서로 다른 정의 3개이며 같은 시드로 재현된다.
- 선택 정의를 같은 런 덱에 고유 ID로 추가할 수 있다.
- 새 런은 최초 덱을 복구한다.
- 기존 CoreLoop·StageProgression·카드 사용 테스트가 통과한다.

### 추천 커밋 제목

`전투 보상을 재현 가능하게 생성하고 런 덱에 안전하게 담을 기반을 마련`

## 6. RW-02 — 보상 진행 상태와 처리

### 목표

승리 뒤 즉시 완료되던 진행을 `RewardSelection`으로 바꾸고 선택·건너뛰기가 끝날 때만 목적 상태로 이동하게 한다.

### 작업

1. `StageProgressionState.RewardSelection`을 추가한다.
2. `BattleRewardCompletionTarget`, `PendingBattleReward`와 최근 결과 모델을 추가한다.
3. `RunProgress`에 보상 시작·선택·건너뛰기 API를 구현한다.
4. 일반 완료와 보스 승리 목적지를 보류 보상에 고정한다.
5. 기존 즉시 완료 API와 테스트를 새 의미에 맞게 마이그레이션한다.
6. 오입력·중복 입력·보상 중 다른 진행 입력을 원자적으로 거절한다.
7. RW02-U01~RW02-U07 테스트를 작성한다.

### 완료 게이트

- 일반과 최종 보스 모두 승리 직후 `RewardSelection`이다.
- 선택과 건너뛰기 뒤에만 각각 `StageCleared`, `RunVictory`가 된다.
- 패배에는 보상이 없다.
- 실패 입력은 상태·덱·다음 ID를 바꾸지 않는다.
- 보상 중 다음 진행과 재시작이 차단된다.

### 추천 커밋 제목

`승리 결과를 보상 선택이 끝난 뒤에만 확정하도록 진행 상태를 연결`

### 구현 결과

- `RewardSelection`, 완료 목적지, 보류 제안과 최근 해결 결과를 추가했다.
- 일반전은 보상 완료 뒤 `StageCleared`, 최종 보스는 높은 등급 보상 완료 뒤 `RunVictory`가 된다.
- 선택은 덱에 카드 한 장만 추가하고 건너뛰기는 덱을 바꾸지 않는다.
- 잘못된 목적지·옵션, 중복 입력과 보상 중 다른 진행 입력은 상태와 덱을 바꾸지 않는다.
- 공개 즉시 완료 API는 제거하고, RW-03 연결 전까지 실제 세션에만 내부 무보상 호환 경계를 남겼다.
- 신규 7/7, StageProgression 49/49와 전체 EditMode 171/171을 통과했다.

## 7. RW-03 — 전투 결과와 세션 통합

### 목표

실제 전투 승리 결과가 알맞은 보상 제안을 한 번 만들고, 선택 카드가 다음 전투 덱에 반영되도록 연결한다.

### 작업

1. `StageProgressionSession`이 승리 시 영혼을 먼저 동기화하게 유지한다.
2. 최종 보스 여부와 상대 보상 등급으로 풀과 완료 목적지를 결정한다.
3. 승리 한 번당 보상 제안을 한 번만 생성한다.
4. 선택·건너뛰기 전달 API를 추가한다.
5. 보상 완료 전 씬 전환이나 다음 전투 생성을 막는다.
6. 선택 카드를 다음 `StageBattleFactory` 덱 구성에 포함한다.
7. 재시작 뒤 이전 획득 카드가 새 전투에 남지 않게 한다.
8. RW03-I01~RW03-I06 테스트를 작성한다.

### 엘리트 경계

현재 스테이지 모델에는 엘리트 타입이 없다. 상대 선택 시스템이 먼저 구현되지 않았다면 테스트용 세션 입력 또는 명시적 `BattleRewardTier` 경계까지만 제공한다. 엘리트를 `StageKind`에 임의 추가하거나 일반 스테이지 인덱스로 추론하지 않는다.

### 완료 게이트

- 일반 승리, 보스 승리와 패배가 각기 올바른 흐름을 사용한다.
- 미래 엘리트 입력을 높은 등급으로 전달할 수 있다.
- 선택 카드는 다음 전투에 포함되고 건너뛰기는 덱을 바꾸지 않는다.
- 런 재시작은 보상 획득 전 상태를 복구한다.

### 추천 커밋 제목

`전투 승리 보상을 다음 전투 덱과 런 결과까지 일관되게 전달`

### 구현 결과

- 승리 영혼을 동기화한 뒤 일반·보스 보상 제안을 한 번만 생성한다.
- 보스는 항상 높은 등급과 런 승리 목적지를 사용한다.
- 일반 스테이지의 등급 결정 함수를 주입해 미래 엘리트의 높은 등급 경계를 제공한다.
- 세션의 선택·건너뛰기 API가 순수 진행 상태에 전달된다.
- 선택 카드는 다음 전투 덱에 포함되고 건너뛰기는 덱 수를 유지한다.
- 재시작은 최초 덱으로 새 전투를 만들며 내부 무보상 호환 경로를 제거했다.
- 신규 6/6, StageProgression 55/55와 전체 EditMode 177/177을 통과했다.

## 8. RW-04 — 진행 화면과 입력

### 목표

기존 `StageTest` 진행 화면에서 후보 3장, 선택과 건너뛰기를 실제로 조작할 수 있게 한다.

### 작업

1. 표시 모델에 보상 등급, 후보 3장, 입력 가능 여부와 완료 안내를 추가한다.
2. Presenter가 정의 키를 카드 이름·숫자·효과 요약으로 변환한다.
3. View에 후보별 `SELECT`와 `SKIP REWARD` 입력을 추가한다.
4. Controller가 입력을 세션에 한 번만 전달한다.
5. 보상 중 전투·다음·재시작 입력을 숨기거나 비활성화한다.
6. 선택 결과, 현재 덱 장수와 보스 완료 안내를 표시한다.
7. RW04-P01~RW04-P05 테스트를 작성한다.
8. 실제 `StageTest` Game View와 Console을 확인한다.

### 완료 게이트

- 후보 3장의 정보와 선택·건너뛰기가 실제 화면에서 동작한다.
- 보스는 보상 전 승리 결과 화면으로 넘어가지 않는다.
- UI가 상대 비공개 카드나 규칙 내부 상태를 노출하지 않는다.
- 스크립트 문제 0, 씬 문제 0, Console Error/Warning 0이다.

### 추천 커밋 제목

`진행 화면에서 전투 보상을 선택하거나 건너뛸 수 있게 연결`

## 9. RW-05 — 반복 회귀와 마감

### 목표

일반·보스 선택과 건너뛰기, 패배와 재시작을 반복 검증하고 전투 보상 1차 범위를 마감한다.

### 작업

2. 후보 정의 고유성, 카드 ID, 덱 소유권과 다음 전투 반영을 검사한다.
3. CoreLoop, StageProgression, 카드 사용과 전체 EditMode를 실행한다.
4. `CoreLoopTest`, `StageTest` 실제 승리·보상·다음 진행·보스·재시작을 확인한다.
5. Console Error/Warning 0을 확인한다.
6. 전투 보상 문서 4종, README, AI 활용 기술서와 이천서 역할 기록을 최종 갱신한다.
7. 신규 패키지·씬·외부 에셋 변경 여부를 기록한다.

### 완료 게이트

- 신규 반복 테스트와 전체 회귀가 모두 통과한다.
- 실제 일반 선택·건너뛰기와 보스 선택·건너뛰기가 설계 순서대로 동작한다.
- 재시작 뒤 획득 카드와 보류 보상이 남지 않는다.
- 문서와 실제 코드·테스트 수치가 일치한다.

### 추천 커밋 제목

`전투 보상과 런 재시작의 반복 안정성을 검증해 1차 범위를 마감`

## 10. 의존 관계

```text
RW-00 문서 기준
  -> RW-01 보상 데이터·생성·덱
      -> RW-02 진행 상태
          -> RW-03 전투·세션 통합
              -> RW-04 화면·입력
                  -> RW-05 전체 검증
```

RW-01의 덱 복구가 검증되기 전에 상태 전이를 연결하지 않는다. RW-02의 원자성이 검증되기 전에 실제 세션을 연결하지 않는다. RW-03이 선택 카드를 다음 전투에 전달하기 전에는 UI만 먼저 만들지 않는다.

## 11. 주요 위험과 대응

| 위험 | 영향 | 대응 |
| --- | --- | --- |
| 기존 즉시 `RunVictory` 테스트와 호출부 충돌 | 보스 보상을 건너뜀 | RW-02에서 상태 테스트와 호출부를 함께 마이그레이션 |
| 읽기 전용 덱을 직접 가변화 | 외부 임의 변경과 재시작 오염 | 내부 가변 목록·외부 읽기 전용 노출·최초 덱 스냅샷 분리 |
| 카드 ID 재사용 | 선택·효과 대상 혼동 | 런 내 단조 증가 발급기와 반복 테스트 |
| 보상 후보 재생성 | 새로고침 악용과 비결정성 | 승리 시 한 번 생성해 보류 상태에 저장 |
| 엘리트 타입 부재 | 높은 등급 연결 불명확 | 명시적 보상 등급 경계를 만들고 상대 선택 작업에서 주입 |
| 보스 카드의 후속 전투 용도 없음 | 체감 보상 약화 | 최종 덱·결과 기록 용도로 명시, 메타 성장은 별도 결정 |

## 12. 커밋 원칙

- RW-00~RW-05를 단계별로 나눈다.
- 제목은 변경 파일이 아니라 결정 목적을 설명한다.
- Lore 본문에 제약, 거절한 대안, 검증과 미검증 항목을 기록한다.
- 자동 생성 `.meta`가 필요한 구현 단계에서는 대응 에셋과 함께 포함한다.
- 사용자가 요청하기 전에는 스테이징하거나 커밋하지 않는다.
