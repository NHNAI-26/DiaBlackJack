## 1. 기록 원칙

- 계획과 실제 구현을 분리한다.
- HONG이 구현하기 전에는 완료 기여로 기록하지 않는다.
- 각 단계마다 작업자, 변경 파일, 직접 결정한 문제, 대상·전체 테스트와 실제 화면 결과를 남긴다.
- AI 초안과 사람의 검토·결정을 구분한다.
- 전투·보상·상대 선택 내부 변경은 이유와 이천서 검토 여부를 기록한다.

| 최근 자동 기준선 | EUI-05 전체 EditMode 315/315 통과 기록 |

## 4. RF-00 수행 기록

### 4.1 수행 내용

- 최신 `rule.md`, 전체 기획, SP·RW·EUI 문서와 현재 `PlayerRunState`, `StageProgressionSession`, Runtime·Controller를 대조했다.
- 작업 중 기준 문서가 갱신된 것을 확인하고 이전 휴식·상점·사건 3종 랜덤 초안을 폐기했다.
- 최신 기준인 전투 승리 골드, 상점 내 유료 휴식, 첫 개발의 두 상점 고정을 반영했다.
- 적별 골드 3·3·4·6·10과 상점 구매 4, 제거 3, 휴식 3골드·영혼 3을 프로토타입 임시값으로 정했다.
- 기존 전투 세션을 감싸는 정식 런 세션과 새 상점 도메인의 책임을 분리했다.
- 이천서를 기획·통합 책임자, HONG을 구현 예정 담당자로 기록했다.

## 6. 인수인계 핵심 계약

1. `StageProgressionSession`은 상대 선택·전투·카드 보상·전투 스테이지를 계속 소유한다.
2. 정식 런 UI는 기존 카드 보상 API를 직접 호출하지 않고 바깥 세션을 사용한다.
3. 카드 선택과 건너뛰기 모두 승리 골드를 한 번 지급한다.
4. 일반전 카드 보상 뒤에는 다음 전투가 아니라 상점을 연다.
5. 상점에서 나간 뒤에만 `TryAdvanceToNextStage()`를 호출한다.
6. 첫 상점 뒤에는 상대 선택, 둘째 상점 뒤에는 고정 보스다.
7. 보스 골드는 기록하지만 상점을 열지 않는다.
8. 상점 변경 영혼·덱은 다음 전투 생성 전에 반영한다.
9. 카드 강화·유물·무작위 사건은 RF에서 선점하지 않는다.

## 7. 위험과 대응

| 위험 | 대응 |
| --- | --- |
| 이전 랜덤 이벤트 기획을 구현 | 첫 개발은 상점 고정이라는 최신 규칙을 테스트 이름과 문서에 명시 |
| 기존 보상 API 직접 호출로 골드 누락 | 정식 UI를 `FormalRunSession`으로 라우팅 |
| 상점 전 다음 전투 생성 | 나가기 승인 뒤에만 기존 다음 단계 API 호출 |
| 비용만 빠지고 효과 실패 | 모든 조건 사전 검증 뒤 비용·효과·플래그 일괄 적용 |
| 화면 재진입으로 상품 재추첨 | `ShopVisit`과 불변 Offer 보존 |
| 제거한 카드 ID 재사용 | 기존 단조 증가 ID를 줄이지 않음 |
| 담당 파일 충돌 | 새 RunFlow·Shop 폴더 우선, 기존 UI 최소 변경, 이천서 검토 |

## 8. 다음 작업

HONG이 먼저 착수할 작업은 **RF-01 런 골드 상태와 전투 승리 정산**이다. 상점 UI나 전체 순서를 동시에 구현하지 않는다. RF-01 대상 테스트와 StageProgression 전체 회귀가 통과한 뒤 RF-02로 넘어간다.
