실제 착수·구현·검증 결과는 `Docs/core-loop-progress-log.md`에 누적한다.

## 1. 계획 목적

`Docs/core-loop-design.md`에서 정의한 최소 플레이 흐름을 `CoreLoopTest` 씬에서 검증 가능한 상태로 구현한다. 이 문서는 기능을 추가로 기획하는 문서가 아니라 작업 순서, 담당, 의존 관계, 완료 증거와 일정을 관리하는 실행 계획이다.

## 2. 구현 목표

아래 흐름을 중단 없이 반복할 수 있는 빌드를 만든다.

`전투 초기화 → 라운드 배분 → 플레이어 히트/스탠드 → 적 행동 → 라운드 판정 → 영혼 피해 → 다음 라운드 또는 전투 종료 → 재시작`

### 2.1 성공 기준

- 플레이어 영혼 12, 적 영혼 3으로 전투를 시작한다.
- 숫자 1~10 각 2장으로 구성된 덱이 정상적으로 뽑히고 재순환한다.
- 에이스, 히트, 스탠드, 버스트와 합계 비교가 규칙대로 처리된다.
- 영혼이 남아 있으면 다음 라운드, 한쪽 영혼이 0이면 결과 화면으로 이동한다.
- 승리와 패배 양쪽에서 재시작할 수 있다.
- 자동 테스트와 10회 연속 수동 재시작에서 진행 불가 오류가 없다.

## 3. 범위 통제

### 3.1 이번 구현에 포함

- 한 전투의 초기화와 종료
- 기본 카드·덱·손패·버린 카드 더미
- 에이스를 포함한 최적 합계 계산
- 플레이어 히트와 스탠드
- `16 이하 히트, 17 이상 스탠드` 단순 적 정책
- 버스트, 합계 비교, 정확히 21과 영혼 피해
- 최소 상태 표시와 입력 UI
- 전투 결과와 재시작
- 규칙 단위 테스트와 코어 흐름 테스트

### 3.2 이번 구현에서 제외

- 카드 특수 효과, 카드 사용, 폴드와 체인지
- 악마 계약과 영혼 소비 선택
- 추론 확률, 적 성향과 고급 AI
- 보상, 덱 빌딩, 경로, 휴식, 상점과 저장
- 다수의 적, 보스 고유 패턴
- 완성 아트, 사운드, 애니메이션과 튜토리얼

제외 기능에 필요한 추상화나 데이터 구조를 미리 만들지 않는다. 후속 확장은 규칙 로직과 UI를 분리하는 현재 경계 안에서 진행한다.

## 4. 작업 원칙

1. 규칙 계산을 먼저 만들고 UI는 검증된 결과만 표시한다.
2. 합계, 피해, 카드 이동과 상태 전이는 각각 한 곳에서만 변경한다.
3. 각 작업은 완료 조건을 충족하고 테스트를 통과한 뒤 다음 작업으로 넘긴다.
4. 새로운 외부 패키지를 추가하지 않는다.
5. 현재 저장소의 공용 Core와 결정적 난수 유틸리티를 우선 검토해 재사용한다.
6. 씬과 Inspector 연결에 의존하지 않는 규칙 로직은 EditMode 테스트 가능하게 작성한다.
7. 계획 담당과 실제 구현 담당을 구분하고 완료 즉시 팀 역할 기술서에 기록한다.

## 6. 작업별 실행 조건

### CL-00 — 기준 확정과 현재 상태 점검

**할 일**

- `rule.md`, 코어 루프 기획서와 개발 명세서의 피해·판정 값을 대조한다.
- 기존 `CoreLoopTest.unity` 씬과 공용 스크립트 중 재사용 가능한 항목을 확인한다.
- 현재 미커밋 변경과 작업 소유자를 확인해 덮어쓰지 않는다.
- 테스트 실행 방법과 목표 씬을 고정한다.

**완료 증거**

- 구현자가 사용할 규칙 값에 미확정 표시가 없다.
- 수정 대상 파일과 건드리지 않을 파일이 작업 메모에 기록되어 있다.

### CL-02 — 카드·덱·합계 기반

**할 일**

- 숫자와 공개 상태만 가진 최소 카드 모델을 만든다.
- 숫자 1~10 각 2장인 기본 덱을 생성하고 결정적으로 섞는다.
- 드로우, 버림, 덱 소진 시 재섞기를 구현한다.
- 여러 에이스가 있는 경우까지 최적 합계를 계산한다.
- 합계와 덱 순환의 테스트를 먼저 작성한다.

**완료 증거**

- `CL-U01`~`CL-U05` 테스트 통과
- 동일 시드에서 동일한 드로우 순서 재현
- 뽑을 덱과 버린 카드 더미의 카드 총수가 보존됨

### CL-04 — 라운드 판정과 영혼 피해

**할 일**

- 버스트, 양측 스탠드, 동점과 정확히 21을 판정한다.
- 판정 결과와 피해 값을 하나의 결과로 반환한다.
- 피해를 한 번만 적용하고 영혼이 0 아래로 표시되지 않게 한다.

**완료 증거**

- `CL-U08`~`CL-U12` 테스트 통과
- UI 없이도 입력 패 조합만으로 결과를 검증 가능

### CL-01 — 전투·라운드 상태 흐름

**할 일**

- 초기화, 라운드 시작, 플레이어 차례, 적 차례, 판정, 전투 종료 상태를 연결한다.
- 각 라운드 시작 시 양측에 공개 1장·비공개 1장을 배분한다.
- 카드 드로우 직후 버스트를 확인한다.
- 판정 중 입력을 차단하고 결과 적용 후 카드 전체를 버린다.

**완료 증거**

- 코드 또는 테스트에서 모든 정상 상태 전이가 한 번 이상 실행됨
- 한 라운드 결과가 두 번 적용되지 않음
- 영혼이 남으면 새 라운드로 자동 전환

### CL-03 — 플레이어 행동과 단순 적 AI

**할 일**

- 플레이어 히트·스탠드 요청을 현재 상태에서만 수락한다.
- 적은 합계 16 이하에서 히트, 17 이상에서 스탠드한다.
- 한쪽만 스탠드하면 반대쪽 행동을 이어 간다.
- 플레이어 비공개 정보가 적 정책 입력에 포함되지 않게 한다.

**완료 증거**

- `CL-U06`, `CL-U07`, `CL-F02`, `CL-F03` 테스트 통과
- 유효하지 않은 상태의 입력은 게임 상태를 바꾸지 않음

### CL-05 — 최소 UI와 입력 연결

**할 일**

- 양측 영혼, 카드, 합계, 덱 수, 현재 상태와 결과를 표시한다.
- 히트·스탠드 버튼을 플레이어 차례에만 활성화한다.
- 입력 직후 버튼을 잠가 중복 처리를 막는다.
- 게임 규칙을 UI 스크립트에서 다시 계산하지 않는다.

**완료 증거**

- `CoreLoopTest` 씬에서 마우스로 한 전투 진행 가능
- 화면 값과 내부 상태가 모든 행동 직후 일치
- 미완성 아트 없이 기본 텍스트·버튼만으로 사용 가능

### CL-06 — 전투 종료와 재시작

**할 일**

- 플레이어와 적 영혼 0을 각각 패배와 승리로 연결한다.
- 결과 상태에서는 전투 입력을 막고 재시작만 허용한다.
- 재시작 시 덱, 버린 카드, 손패, 영혼, 상태와 UI를 모두 새로 초기화한다.
- 이전 이벤트·코루틴이 새 전투에 영향을 주지 않게 정리한다.

**완료 증거**

- `CL-F04`~`CL-F06` 테스트 통과
- 승리·패배 양쪽에서 초기 상태로 복귀
- 연속 10회 재시작 중 중복 이벤트와 진행 정지 없음

### CL-07 — 통합 검증과 기록

**할 일**

- 전체 EditMode 테스트를 실행한다.
- `CoreLoopTest` 씬에서 승리·패배·재시작을 수동 확인한다.
- Unity Console의 Error와 Exception을 확인한다.
- 실제 구현 담당, 파일, 기술적 결정과 테스트 결과를 문서에 기록한다.

**완료 증거**

- 알려진 테스트 실패와 코어 루프 오류 0건
- 완료 조건 체크리스트와 팀원별 실제 기여 기록 갱신
- AI가 구현을 보조한 경우 AI 활용 기록 갱신

## 7. 일정

### 1일차 — 규칙 기반

- 오전: CL-00 기준 확정과 기존 상태 점검
- 오전~오후: CL-02 카드·덱·합계 테스트 및 구현
- 오후: CL-04 판정·피해 테스트 및 구현
- 종료 조건: UI 없이 카드 합계와 모든 기본 판정 검증 가능

### 2일차 — 상태와 행동

- 오전~오후: CL-01 전투·라운드 상태 흐름
- 오후: CL-03 히트·스탠드와 단순 적 AI
- 종료 조건: 테스트 또는 로그만으로 여러 라운드가 끝까지 반복됨

### 3일차 — 화면 통합

- 오전~오후: CL-05 최소 UI와 입력 연결
- 오후: CL-06 승리·패배와 재시작
- 종료 조건: `CoreLoopTest` 씬에서 한 전투를 직접 플레이하고 재시작 가능

### 4일차 — 검증과 안정화

- 오전: 누락 테스트, 경계 조건과 회귀 수정
- 오후: 승리·패배 각각 반복 플레이, 10회 재시작
- 종료 전: 팀 역할 기술서와 AI 활용 기술서 갱신
- 종료 조건: 완료 정의 충족 및 알려진 진행 불가 오류 0건

작업이 늦어지면 UI 연출과 코드 확장성을 줄인다. 핵심 규칙, 상태 흐름과 테스트는 축소하지 않는다.

## 8. 병렬 작업 방식

팀원 배정이 완료되면 다음 작업만 병렬로 진행한다.

- CL-02와 CL-04 담당은 순수 규칙과 테스트를 소유한다.
- CL-01 담당 이천서는 규칙 인터페이스가 확정된 뒤 상태 흐름을 통합한다.
- CL-05 담당은 읽기 전용 화면 계약과 임시 UI 구성을 먼저 준비할 수 있지만 규칙을 자체 구현하지 않는다.

공유 씬과 조립 파일은 한 시점에 한 명만 수정한다. 다른 담당자는 프리팹, 스크립트 또는 테스트처럼 충돌이 적은 파일에서 작업하고 통합 담당자에게 전달한다.
