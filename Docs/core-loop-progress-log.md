## 1. 문서 목적

코어 루프를 단계별로 준비, 구현, 검증한 결과를 한 문서에 누적한다. 전체 범위와 구조를 다시 설명하지 않고 각 단계에서 실제로 수행한 작업, 계획과 달라진 내용, 테스트 결과와 담당자를 기록한다.

기준 문서는 다음과 같다.

- `Docs/core-loop-design.md`: 구현 대상과 제외 범위
- `Docs/core-loop-development-spec.md`: 규칙, 상태와 테스트 명세
- `Docs/core-loop-implementation-plan.md`: 작업 순서, 일정과 검증 게이트

## 2. 기록 방법

각 단계는 다음 순서로 갱신한다.

1. 단계 시작 전에 목표, 담당자, 예정 파일과 테스트를 확인한다.
2. 상태를 `착수 전`에서 `진행 중`으로 바꾸고 착수 날짜를 적는다.
3. 구현 중 발생한 중요한 변경만 결정 기록에 추가한다.
4. 테스트 결과와 실제 구현 파일을 기록한다.
5. 완료 조건을 모두 확인한 뒤 상태를 `완료`로 바꾼다.
6. `team-role-technical-document.md`와 `ai-usage-technical-document.md`를 갱신한다.
7. 다음 단계의 세부 항목을 직전 단계 결과에 맞게 보정한다.

### 2.1 상태 정의

| 상태 | 의미 |
| --- | --- |
| 착수 전 | 계획만 존재하며 구현을 시작하지 않음 |
| 진행 중 | 담당자가 코드를 작성하거나 테스트 중 |
| 검증 중 | 구현은 끝났으며 테스트와 수동 확인 중 |
| 완료 | 완료 조건과 검증 게이트를 모두 통과 |
| 보류 | 외부 결정이나 선행 작업 때문에 일시 중단 |

단순히 코드를 작성한 상태는 완료가 아니다. 테스트 결과와 실제 담당 기록까지 있어야 완료로 처리한다.
