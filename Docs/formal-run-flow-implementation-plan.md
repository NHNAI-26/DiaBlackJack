## 1. 실행 원칙

HONG은 `RF-01 → RF-05`를 순서대로 구현한다. 각 단계에서 대상 테스트와 전체 회귀를 통과하고 진행 기록을 갱신한 뒤 다음 단계로 넘어간다. 이천서는 기존 전투·보상·상대 선택 경계 변경과 최종 통합을 검토한다.

## 3. RF-00 — 문서와 인수인계

### 작업·완료 증거

- 최신 `rule.md`의 골드·상점 고정 구조와 현재 코드를 대조한다.
- 이천서와 HONG의 소유 파일·변경 경계를 기록한다.
- 적별 골드와 상점 가격·회복량을 프로토타입 임시값으로 분리한다.
- 문서 4종, README, AI 활용과 팀 역할 계획 기록을 작성한다.
- 코드·테스트·씬·패키지·외부 에셋은 변경하지 않는다.

### 권장 커밋 제목

`정식 런 진행 분업의 구현 기준을 확정하다`

## 4. RF-01 — 런 골드와 승리 정산

### 작업

- `PlayerRunState`에 호환 가능한 골드 보유·지급·소비·재시작 경계를 추가한다.
- `GoldRewardCatalog`에 다섯 프로필의 3·3·4·6·10 값을 등록한다.
- 카드 보상 성공 뒤 한 번만 골드를 지급하는 정산 경계를 만든다.
- 카드 건너뛰기, 보스 보상, 패배와 잘못된 입력을 구분한다.
- UI·상점·씬은 아직 변경하지 않는다.

### 검증 게이트

- 프로필별 지급량과 엘리트 우위가 테스트로 고정된다.
- 선택/건너뛰기는 지급하고 실패/중복/패배는 지급하지 않는다.
- 재시작 골드는 0이다.
- 기존 전체 테스트가 통과한다.

### 권장 커밋 제목

`위험한 상대 선택이 골드 보상으로 이어지게 하다`

## 5. RF-02 — 상점 도메인

### 작업

- 불변 `ShopCardOption`, `ShopOffer`, 거래 결과와 결정적 생성기를 구현한다.
- 일반 보상 풀에서 서로 다른 구매 후보 3장을 만든다.
- 카드 구매 4골드, 카드 제거 3골드, 휴식 3골드·영혼 3 회복을 구현한다.
- 각 서비스 방문당 1회와 무료 나가기를 구현한다.
- `PlayerRunState`의 카드 제거는 덱 최소 1장과 ID 미재사용을 지킨다.
- Unity UI와 정식 순서는 아직 연결하지 않는다.

### 검증 게이트

- 거래 성공은 비용과 효과를 한 번에 반영한다.
- 부족한 골드·잘못된 대상·중복 이용은 모든 상태를 보존한다.
- 제안은 같은 시드에서 결정적이고 재진입으로 바뀌지 않는다.
- 기존 전체 테스트가 통과한다.

### 권장 커밋 제목

`골드를 성장 선택으로 바꿀 상점 거래를 마련하다`

## 6. RF-03 — 정식 순서 통합

### 작업

- `FormalRunPhase`와 `FormalRunSession`을 구현한다.
- 기존 상대 선택·전투·카드 보상은 `StageProgressionSession`에 위임한다.
- 일반전 보상 완료 뒤 골드를 정산하고 상점을 연다.
- 첫 상점 나가기 뒤 두 번째 상대 선택, 둘째 상점 뒤 고정 보스로 이동한다.
- 보스 보상은 골드를 정산하고 상점 없이 런 승리로 마친다.
- 패배와 재시작에서 모든 후속·이전 상태를 차단·초기화한다.

### 검증 게이트

- `전투 보상·골드 → 상점 → 전투 보상·골드 → 상점 → 보스` 순서가 통합 테스트로 고정된다.
- 상점에서 나가기 전 스테이지 인덱스가 증가하지 않는다.
- 상점 변경 영혼·덱이 다음 전투에 반영된다.
- 기존 고정 진행 세션의 회귀가 유지된다.

### 권장 커밋 제목

`전투와 보스 사이에 두 번의 상점 흐름을 고정하다`

## 7. RF-04 — 진행 화면과 씬 연결

### 작업

- Runtime이 정식 런 세션을 소유하고 전투 Controller에는 내부 전투 세션을 제공한다.
- Presenter에 골드·상점 후보·가격·가능 여부·최근 거래를 추가한다.
- View와 Controller에 구매·제거·휴식·나가기 입력을 연결한다.
- 상점 중 상대·보상·NEXT STAGE·재시작 입력을 차단한다.
- 첫 상점 뒤 상대 선택, 둘째 상점 뒤 보스 씬 이동을 연결한다.

### 검증 게이트

- 카드 선택/건너뛰기 뒤 골드와 상점이 표시된다.
- 거래 가능/불가 상태와 결과가 실제 데이터와 일치한다.
- 1280×720·1920×1080에서 겹침·잘림이 없다.
- 실제 `StageTest ↔ CoreLoopTest` 흐름과 Console이 통과한다.

### 권장 커밋 제목

`정식 런의 골드와 상점을 기존 진행 화면에 잇다`

## 8. RF-05 — 반복 검증과 마감

### 작업

- 전체 승리, 세 전투 위치의 패배, 선택/건너뛰기와 재시작을 반복한다.
- 네 적 프로필 선택과 적별 골드, 상점 세 서비스를 모두 검증한다.
- 오래된 상대·보상·상점 ID와 중복 입력의 상태 격리를 검증한다.
- 전체 EditMode, 실제 두 씬, 두 해상도와 Console을 최종 확인한다.
- HONG의 실제 변경 파일·결정·테스트를 진행 기록과 팀 역할 문서에 추가한다.

### 반복 기준

| 흐름 | 반복 |
| --- | ---: |
| 전체 정식 런 승리·재시작 | 10회 |
| 구매·제거·휴식·나가기 | 각 10회 |
| 오래된·중복 입력 거부 | 10회 |
| 첫째·둘째·보스 패배·재시작 | 각 10회 |

### 완료 게이트

- 신규 RF와 전체 기존 테스트가 모두 통과한다.
- 실제 흐름·골드·상점이 문서와 일치한다.
- 게임 관련 Console Error·Exception이 0이다.
- HONG의 계획 역할이 실제 기여 증거로 갱신된다.
- 이천서가 최종 통합을 검토한다.

### 권장 커밋 제목

`정식 런의 골드와 상점 흐름을 반복 검증하다`

## 9. 변경 제한

- 전투 판정, 적 AI, 카드 효과와 보상 풀은 RF에서 수정하지 않는다.
- 골드 지급량은 새 카탈로그에 두고 적 프로필 생성자에 억지 필드를 추가하지 않는다.
- 상점 카드 구매는 기존 일반 보상 정의를 읽되 전투 보상 상태를 재사용하지 않는다.
- 카드 강화·유물·무작위 사건을 빈 구조로 미리 만들지 않는다.
- 다른 팀원의 미완료 변경을 되돌리지 않는다.
- 기존 공개 API 변경이 필요하면 진행 기록에 이유·대안·이천서 검토를 남긴다.
