## 4. 현재 코드와 미구현 영역

### 4.1 이미 가능한 것

- 프로필 안전 미리보기 조회
- 키 하나로 실제 전용 적 덱·영혼·정책 생성
- 엘리트·보스 높은 등급 보상
- 보스 구간·예고 표시 모델 생성
- 진행 씬과 전투 씬 왕복
- 결정적 후보 2명 생성과 0%·100% 엘리트 구성
- 후보 제안 불변성·중복/보스/엘리트 2명 차단
- `OpponentSelection` 진행 상태와 선택 기능 세션 주입
- 일반 시작·다음 일반 스테이지·재시작의 새 제안, 보스 선택 우회
- 후보 2명의 이름·등급·최대 영혼·성향·예상 보상 비교 화면
- Controller 로컬 집중과 제안 밖 키·대소문자 불일치 차단
- 선택 상태에서 `StageTest`를 유지하는 시작·다음 스테이지 라우팅

### 4.2 아직 구현되지 않은 것

- 선택된 프로필을 현재 스테이지에서 확정하는 세션 API
- 확정 입력의 세션 처리와 선택 성공 뒤 전투 씬 이동
- 일반 적 추론 확률 표시 모델
- 기존 엘리트·보스 표시 모델의 실제 전투 View 연결
- 상대 선택→전투→보상→다음 선택 반복 흐름

### 계획과 실제 차이

- 최초 분업안에서는 HONG이 상대 선택·런 진행을 함께 담당할 예정이었다.
- HONG이 해당 게임 코드에 착수하지 않은 상태에서 이천서가 상대 선택·전투 UI 통합까지 담당하기로 합의했다.
- 랜덤 이벤트와 정식 런 진행 확장은 HONG에게 남겨 작업 집중을 방지했다.
- 전투 정보 UI는 EP-06 전투 변환과 분리된 별도 EUI 작업으로 관리한다.

### 남은 위험

- `OpponentSelection`을 기존 `RunProgress` 상태와 연결할 때 진행 테스트 수정 범위가 커질 수 있다.
- 일반 추론 확률 UI는 현재 공개 표시 타입이 없어 안전한 스냅샷이 추가로 필요하다.
- 보스 `CurrentDisplay`를 읽기 위해 정책 객체를 UI에 직접 노출하면 결합도가 높아질 수 있으므로 Factory 경계를 우선한다.
- 두 후보 카드와 기존 보상 UI가 같은 View를 사용하므로 작은 해상도에서 세로 공간을 검증해야 한다.
- 엘리트 35%는 밸런스 로그 전의 임시 값이다.

- 계획의 예상 파일에는 Runtime이 없었으나 실제 선택 화면을 프로토타입에서 열려면 선택 생성기 주입이 필요해 `StageProgressionRuntime`을 함께 수정했다.

- EditMode에서 Runtime `Awake`를 직접 검증하는 최초 테스트는 `DontDestroyOnLoad`의 Play Mode 제한으로 실패해 제거하고, 실제 Play Mode 화면 검증으로 Runtime 활성화를 확인했다.

### 남은 위험

- 확정 버튼은 유효한 집중 뒤 활성화되지만 EUI-03 전까지 세션 확정과 전투 씬 이동을 수행하지 않는다.
- 현재 후보 이름·요약은 프로필 원문을 사용하므로 영문 대문자 중심의 기존 진행 UI와 한글이 섞인다. 정식 현지화는 별도 범위다.
