## 1. 기록 원칙

- 계획, 착수, 구현, 검증과 완료를 구분한다.
- 실행하지 않은 테스트를 통과로 기록하지 않는다.
- 단계별 코드·씬·문서 변경과 이천서의 실제 담당 범위를 남긴다.
- AI 대화 원문이 아니라 목적, 주요 지시, 결과와 사람의 검토 상태를 정리한다.
- 임시 기획 결정이 바뀌면 이유와 영향 단계를 함께 기록한다.

## 5. 단계별 기록

### BA-00 — 구현 기준 확정

**상태:** 완료
**담당:** 이천서
**AI 역할:** 기준 문서·코드 구조 대조, 기획 충돌 정리, 문서 초안과 자체 점검 보조

#### 수행 내용

- `rule.md`와 `game-design-document.md`의 행동 규칙을 대조했다.
- 현재 `CoreLoopBattle`, 세션, 덱·손패, UI와 진행 세션의 전달 구조를 확인했다.
- 폴드·체인지 포함 범위와 카드 사용·계약 제외 범위를 정했다.
- 체인지의 선택 상태, 후보 정보와 카드 위치 불변 조건을 명세했다.
- BA-00~BA-05 구현·검증·커밋 순서를 정했다.

#### 산출물

- `Docs/combat-action-design.md`
- `Docs/combat-action-development-spec.md`
- `Docs/combat-action-implementation-plan.md`
- `Docs/combat-action-progress-log.md`

#### 검증

- 문서 간 단계 ID, 범위와 용어 대조
- 신규 행동 코드가 아직 구현되지 않았음을 명시
- 기존 미커밋 문서 변경을 덮어쓰지 않고 추가 기록만 수행
- 코드·씬·Unity 테스트는 문서 작성 단계이므로 실행하지 않음

#### 다음 단계 진입 조건

- BA-01 착수 시 현재 Git 상태 확인
- Unity 프로젝트·MCP 연결 확인
- 기존 전체 EditMode 테스트 재실행
- 실패가 있으면 신규 구현 전 기준선 문제로 분리

### BA-01 — 상태·카드 이동 기반

**상태:** 완료
**담당:** 이천서
**AI 역할:** 현재 코드·테스트 구조 분석, 기반 코드·테스트 초안, Unity MCP 컴파일·회귀 검증과 기록 보조

#### 수행 내용

- `CoreLoopState.PlayerChoosingChangeCard`를 추가했다.
- `BlackjackHand`가 비공개 카드 수를 제공하고, 정확히 한 장일 때만 상태 변화 없이 검증 후 인출하도록 했다.
- `BlackjackDeck`이 뽑을 더미와 버린 더미에서 확보 가능한 카드 수를 계산하고 `CanDraw`로 검사하도록 했다.
- `PlayerChangeSelection`이 기존 비공개 카드와 후보 2장의 ID 중복을 거부하고, 후보 선택을 한 번만 완료하도록 했다.
- 잘못된 후보 인덱스, 비공개 카드 0장·복수 장, 후보 부족 검사의 무변경 조건을 테스트했다.
- 범용 행동 디스패처, UI, 전투 행동 API와 진행 세션 연결은 추가하지 않았다.

#### 변경 파일

- `Assets/01. Scripts/Runtime/CoreLoop/CoreLoopState.cs`
- `Assets/01. Scripts/Runtime/CoreLoop/BlackjackHand.cs`
- `Assets/01. Scripts/Runtime/CoreLoop/BlackjackDeck.cs`
- `Assets/01. Scripts/Runtime/CoreLoop/PlayerChangeSelection.cs`
- `Assets/Tests/EditMode/CoreLoop/CombatActionFoundationTests.cs`
- 관련 `.meta` 파일과 진행·AI·팀 역할·프로젝트 구조 문서

#### 검증

- 구현 전 전체 EditMode 50/50 통과, job `4ad5b78d5861437b8d22f89a3a34957e`
- Unity 재컴파일 성공, 컴파일 오류 0
- CoreLoop 35/35 통과, job `05b3d109af5e46d58c9922ffc7c08691`
- 전체 EditMode 58/58 통과, job `27be56769b124d0b84a9c4016547cbea`
- Console의 6개 메시지는 Test Framework 사전·사후 처리와 결과 저장 기반 시설 출력이며 게임 오류는 0

#### 계획 조정

라운드당 체인지 사용 여부는 BA-01에서 별도 미사용 필드로 만들지 않았다. BA-03에서 `CoreLoopBattle`의 체인지 시작·완료와 새 라운드 초기화가 실제로 연결될 때 추가하고 같은 테스트 흐름으로 검증한다.

## 8. 미해결 사항

- 체인지 후보 표시를 현재 IMGUI 화면에서 카드 버튼으로 표현할 세부 배치는 BA-04에서 화면을 보고 확정한다.
- 향후 복수 비공개 카드 효과가 추가되면 `비공개 카드 정확히 1장` 전제는 다시 검토한다.
- 적의 폴드·체인지 사용 여부는 적 AI 작업에서 별도로 결정한다.
- 카드 사용과 계약의 공통 선택 처리 구조는 각 기획 문서 작성 전까지 만들지 않는다.
