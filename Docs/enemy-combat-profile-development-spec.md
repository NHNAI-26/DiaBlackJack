# 적 전투 프로필 시스템 개발 명세서

> 프로젝트: DiaBlackJack  
> 기획·개발 책임자: 이천서  
> 버전: v0.6
> 상태: EP-06 선택 전투 통합·반복 검증 완료, 1차 범위 마감
> 최종 갱신: 2026-07-20

## 1. 기술 목표

현재의 고정 `SimpleEnemyPolicy`를 기존 전투 호환성을 유지하는 교체 가능 정책 경계로 확장하고, 안정적인 프로필 키 하나로 적 영혼·덱·행동 정책·표시 미리보기·보상 등급을 일관되게 구성한다.

상대 선택 시스템은 프로필 키만 선택하고, CoreLoop는 공개 관측 상태와 적 정책으로 행동을 결정한다. 진행·전투 연결부는 선택 결과를 전투 설정으로 변환하되 상대 선택 규칙을 다시 계산하지 않는다.

## 2. 현재 코드 기준

2026-07-20 EP-00 작성 시점은 다음과 같다.

- `CoreLoopBattle`은 구체 타입 `SimpleEnemyPolicy`를 직접 보유한다.
- `SimpleEnemyPolicy.Decide(HandValue)`는 적 자신의 합계가 16 이하면 히트, 17 이상이면 스탠드를 반환한다.
- 적 차례는 히트·스탠드만 반복하며 적 카드 사용·폴드·체인지·계약 경계가 없다.
- `CardEffectResolver`에는 수정 구슬, 위협용 해머, 리볼버과 보위 나이프 처리기가 등록되어 있다.
- 현재 카드 카탈로그에는 거짓말 탐지기와 악마 계약 카드가 없다.
- `StageDefinition`은 기존 직접 최대 영혼 생성 경로와 프로필 키 기반 생성 경로를 함께 제공한다.
- `StageBattleFactory.Create(StageDefinition, PlayerRunState)`는 프로필 키가 있으면 전용 적 덱·최대 영혼·새 행동 정책을 사용하고, 키가 없으면 기존 표준 덱과 기본 정책을 유지한다.
- `StageProgressionSession`은 전투 생성 함수와 보상 등급 선택 함수를 주입받을 수 있다.
- `StageProgressionRuntime`은 일반전 2개와 보스 1개를 코드에 하드코딩한다.
- 상대 후보·선택 결과·적 프로필 카탈로그는 아직 없다.

따라서 EP 구현은 CoreLoop의 정책 인터페이스, 공개 관측 모델과 프로필 기반 전투 설정을 먼저 만들고, 상대 선택·진행 통합은 안정된 프로필 키 경계 뒤에 연결해야 한다.

## 3. 설계 원칙

1. 기존 CoreLoop 상태와 플레이어 행동 규칙을 다시 작성하지 않는다.
2. 적 정책은 전투 객체 전체가 아니라 허용된 공개 관측 데이터만 받는다.
3. 프로필 데이터와 행동 알고리즘을 분리한다.
4. 상대 선택 화면과 실제 전투 설정은 같은 프로필 카탈로그를 기준으로 한다.
5. 적 행동 결정과 실행을 분리해 잘못된 결정을 원자적으로 거절한다.
6. 기존 `SimpleEnemyPolicy`의 기본 동작과 생성자 호환을 보존한다.
7. 구현되지 않은 카드·계약을 가짜 결과로 대체하지 않는다.
8. 동일 시드·관측 상태에서 결정과 전투 덱을 재현할 수 있게 한다.
9. 새 외부 패키지나 AI 네트워크 서비스를 추가하지 않는다.

## 5. 데이터 타입

이름은 코드 규칙에 따라 조정할 수 있으나 책임은 유지한다.

### 5.1 `EnemyGrade`

```text
Normal
Elite
Boss
```

알 수 없는 열거값은 생성 시 거절한다.

### 5.2 `EnemyCombatProfile`

필수 필드:

- `Key`: 저장·선택·테스트에서 사용하는 안정적인 문자열 키
- `DisplayName`: 플레이어 표시 이름
- `Grade`: 일반·엘리트·보스
- `MaximumSoul`: 양수
- `BehaviorPolicyKey`: 정책 카탈로그 키
- `DeckDefinitionKeys`: 적 덱의 카드 정의 키 목록
- `Summary`: 선택 화면에 표시할 핵심 성향
- `PlayerInformationMode`: 일반·엘리트·보스 추론 표시 수준

생성 시 빈 키·이름·정책, 잘못된 등급·영혼, 빈 덱, 존재하지 않는 카드 정의와 중복 정책 등록을 거절한다.

### 5.3 `EnemyProfilePreview`

상대 선택 시스템에 제공하는 읽기 전용 값이다.

- 프로필 키
- 표시 이름
- 등급
- 최대 영혼
- 성향 요약
- 예상 보상 등급

덱 순서, 정책 객체, 행동 가중치와 비공개 정보는 포함하지 않는다.

### 5.4 `EnemyObservation`

정책에 전달할 불변 스냅샷이다.

- 적 자신의 손패 공개·비공개 전체 정보
- 플레이어의 공개 카드만
- 공개·버린 카드의 정의 또는 숫자 집계
- 양측 현재·최대 영혼
- 현재 라운드와 공개 행동 이력
- 적이 소유한 사용 가능한 카드와 승인 전 가용성
- 공개된 활성 효과
- 규칙적으로 계산한 추론 분포

플레이어 비공개 카드 객체, 실제 숫자, 물리 ID와 양쪽 덱 순서는 타입 자체에 넣지 않는다.

EP-03의 `NumberInferences`는 플레이어 덱의 순서 없는 전체 숫자별 구성에서 현재 공개 손패와 공개 버림패를 제외한 후보군으로 계산한다. 확률은 정수 백분율이며 합계를 100으로 보정한 뒤 확률 내림차순·숫자 오름차순으로 정렬한다. 계산 과정에서도 실제 비공개 카드 객체를 읽지 않는다.

### 5.5 `EnemyActionType`

1차 목표 후보:

```text
Hit
Stand
Fold
UseCard
```

`Change`와 `Contract`는 실제 적용 API가 구현된 뒤 추가한다. 열거값만 미리 추가해 항상 실패하는 행동을 만들지 않는다.

### 5.6 `EnemyDecision`

- 행동 종류
- 선택 카드 ID 또는 정의된 선택값
- 카드 효과 옵션
- 결정 사유 코드
- 디버그용 후보 점수 목록

UI에 직접 노출되는 객체가 아니며 비공개 추론 원본을 포함하지 않는다.

### 5.7 `EnemyActionCandidate`

정책이 평가하고 실행기가 승인하는 불변 후보다.

- 공개 행동 종류: 히트·스탠드·폴드·카드 사용
- 카드 사용이면 적이 소유한 물리 카드 ID와 정의
- 선택형 효과면 현재 효과가 공개한 옵션 ID
- 행동·카드·옵션의 정확한 일치 여부

후보 생성은 상태를 변경하지 않는다. 정책이 임의로 만든 카드 ID나 옵션은 실행 직전 `EnemyDecisionValidator`에서 거절한다.

EP-03부터 선택형 카드 후보는 적이 규칙적으로 확인할 수 있는 옵션 정보도 가진다.

- 리볼버: 선언 숫자
- 수정 구슬: 적이 확인한 임시 카드 ID와 숫자

플레이어 비공개 카드 값이나 플레이어 덱의 다음 카드는 옵션 정보에 포함하지 않는다.

## 6. 정책 인터페이스

목표 인터페이스는 다음 책임을 가진다.

```csharp
public interface IEnemyBehaviorPolicy
{
    EnemyDecision Decide(EnemyObservation observation);
}
```

- 호출마다 전투 상태를 변경하지 않는다.
- 같은 관측과 같은 정책 난수 상태에서 같은 결정을 반환한다.
- 실행 불가능한 행동을 반환하지 않도록 승인 전 가용성을 사용한다.
- 구현이 반환한 결정은 적 행동 실행기에서 다시 검증한다.

### 6.1 기존 정책 호환

`SimpleEnemyPolicy`는 바로 삭제하지 않는다.

- 기존 16 이하 히트·17 이상 스탠드 규칙을 보존한다.
- `IEnemyBehaviorPolicy` 어댑터 또는 직접 구현으로 이전한다.
- 기존 생성자와 테스트가 기본 정책을 계속 사용할 수 있게 한다.
- 모든 적 프로필 연결이 끝난 뒤에도 프로필 미지정 테스트의 기본값으로 유지할 수 있다.

### 6.2 EP-03 일반 적 정책

| 정책 키 | 구현 타입 | 1차 기준 |
| --- | --- | --- |
| `gunslinger-public-inference` | `GunslingerEnemyPolicy` | 최상위 숫자 확률 50% 이상이면 리볼버, 아니면 16/17 기본 판단 |
| `cultist-aggressive-risk` | `CultistEnemyPolicy` | 합계 18까지 히트, 19부터 스탠드 |
| `trickster-information-control` | `TricksterEnemyPolicy` | 수정 구슬 우선, 버스트하지 않는 최고 합계 옵션 |

`EnemyPolicyDecisionSelector`가 유효 후보 순회, 점수·사유 수집과 최고 점수 선택을 공통 처리한다. 동일 점수면 관측에 들어온 안정적인 후보 순서를 유지한다. 악마 계약과 거짓말 탐지기 키는 등록하지 않는다.

## 7. 행동 평가 구조

정책은 다음 순서로 동작한다.

1. 현재 상태에서 실행 가능한 후보를 수집한다.
2. 각 후보의 즉시 규칙 결과를 평가한다.
3. 적별 위험·생존·카드·정보 가중치를 적용한다.
4. 엘리트·보스만 필요한 경우 카드 후속 상태를 한 단계 더 평가한다.
5. 결정적 변동값과 동점 우선순위를 적용한다.
6. 최고 점수 후보와 디버그 사유를 반환한다.

점수는 비교 용도이며 게임 결과를 직접 조작하지 않는다.

## 8. 카드 효과 연결

### 8.1 공통 규칙

- 적은 적이 실제 소유한 카드만 사용할 수 있다.
- 카드 가용성 조회는 상태를 바꾸지 않는다.
- 승인된 카드만 공개·사용 상태로 전환한다.
- 적 사용 카드의 행위자는 적, 효과 대상은 카드 정의 규칙에 따른 상대 또는 자신이다.
- 플레이어용 공개 입력 메서드를 적이 호출하는 방식으로 우회하지 않는다.
- 효과가 라운드나 전투를 끝내면 추가 적 행동을 실행하지 않는다.
- 선택형 효과는 정책이 유효한 옵션을 결정한 뒤 기존 효과 완료 경계를 재사용한다.

### 8.2 프로필별 1차 카드

| 프로필 | 카드 | 현재 구현 상태 | EP 처리 |
| --- | --- | --- | --- |
| 총잡이 | 리볼버 | 처리기 존재 | 적 추측 입력·정보 은닉 검증 후 사용 |
| 광신도 | 악마 계약 | 미구현 | 후보에서 제외, 공격 성향만 구현 |
| 사기꾼 | 수정 구슬 | 처리기 존재 | 적 선택 경계와 덱 소유권 검증 후 사용 |
| 사기꾼 | 거짓말 탐지기 | 미구현 | 1차 범위 제외 |
| 집행관 | 위협용 해머·보위 나이프 | 처리기 존재 | 적 행위자·플레이어 대상 방향 검증 후 사용 |
| 최종 보스 | 구현된 핵심 카드 조합 | 일부 존재 | 영혼 구간에 따라 가중치 조정 |

기존 처리기가 플레이어 행위자를 가정한다면 억지로 재사용하지 않고 공통 효과 명령과 행위자 컨텍스트를 먼저 분리한다.

EP-02에서는 `CardEffectContext`를 `Actor`와 `Opponent` 기준으로 일반화했다. 기존 플레이어 공개 API와 처리기는 같은 경계를 계속 사용하며, 적 카드 실행만 별도 복제하지 않는다. 적 선택형 효과는 새 UI 상태 없이 `EnemyTurn` 안에서 유효 옵션 후보를 정책이 선택한다.

## 9. 프로필 카탈로그

`EnemyCombatProfileCatalog`는 다섯 프로필을 명시적으로 등록한다.

- 키 비교는 ordinal을 사용한다.
- 읽기 전용 전체 목록과 키 조회를 제공한다.
- 같은 키 중복을 금지한다.
- 표시 미리보기는 프로필에서 파생한다.
- 상대 선택 시스템이 임의 영혼·등급·보상 값을 덮어쓰지 못하게 한다.
- 프로토타입에서는 ScriptableObject 없이 순수 C# 카탈로그로 시작할 수 있다.

정식 콘텐츠 편집 요구가 생기기 전까지 새 데이터 프레임워크를 도입하지 않는다.

## 10. 전투 생성 변환 경계

목표 흐름은 다음과 같다.

```text
Selected BattleProfileKey
  → EnemyCombatProfileCatalog.GetByKey
  → EnemyBattleConfiguration
  → player run deck + enemy profile deck
  → CoreLoopBattle
```

`EnemyBattleConfiguration`은 다음을 보존한다.

- 프로필 키와 등급
- 적 최대 영혼
- 적 덱 카드 정의
- 적 덱 시드
- 행동 정책
- 예상 보상 등급

전투 생성은 모든 입력을 검증한 뒤 한 번에 수행한다. 잘못된 프로필이면 플레이어 덱과 진행 상태를 변경하지 않는다.

## 11. StageProgression 연결

### 11.1 입력

상대 선택 담당이 전달하는 최소 입력은 `BattleProfileKey`다. 표시용 복제 데이터나 정책 인스턴스를 전달하지 않는다.

### 11.2 전투 생성

`StageDefinition.CreateForEnemyProfile(...)`가 프로필 미리보기에서 최대 영혼을 파생하고 일반/엘리트와 보스 스테이지 종류의 불일치를 거절한다. `StageBattleFactory`는 저장된 키를 `EnemyBattleConfigurationFactory`에 전달해 전용 적 덱과 매 전투 새 정책 인스턴스로 `CoreLoopBattle`을 만든다. 프로필 키가 없는 기존 스테이지는 표준 덱과 `SimpleEnemyPolicy`를 계속 사용한다.

### 11.3 보상 연결

| 적 등급 | 보상 등급 | 완료 목적지 |
| --- | --- | --- |
| Normal | `BattleRewardTier.Normal` | `StageCleared` |
| Elite | `BattleRewardTier.HighGrade` | `StageCleared` |
| Boss | `BattleRewardTier.HighGrade` | 보상 처리 뒤 `RunVictory` |

최종 보스 강제 높은 등급 규칙은 기존 전투 보상 시스템을 유지한다.

일반·엘리트의 기본 보상 등급은 `StageProgressionSession`이 `BattleProfileKey`로 안전 미리보기를 조회해 결정한다. 따라서 선택 담당이 보상 등급을 별도로 전달하거나 덮어쓰지 않는다.

## 12. 상태와 실패 원자성

- AI 결정 조회는 전투 상태를 변경하지 않는다.
- 행동 실행은 적 차례에서만 승인한다.
- 잘못된 카드 ID·옵션·행동 종류는 손패·덱·영혼·상태를 바꾸지 않는다.
- 정책 예외가 발생해도 진행 세션에 전투 결과를 전달하지 않는다.
- 안전 재평가는 최대 한 번만 수행한다.
- 기본 스탠드는 실제로 스탠드 가능한 상태에서만 사용한다.
- 전투가 끝난 뒤 정책 호출과 카드 사용을 거절한다.
- 프로필 키가 없는 전투는 기존 기본 정책으로만 명시적으로 생성한다. 선택된 상대 키의 조회 실패를 기본 적으로 숨기지 않는다.
- 정책이 연속 두 번 무효 결정을 반환하면 스탠드, 스탠드가 불가능하면 첫 유효 후보를 선택한다.
- 폴드는 양측 공통 행동으로 취급하며 적 폴드도 적 영혼 1을 소모하고 라운드를 종료한다. 원문에 적 비용이 별도로 명시되지 않아 첫 밸런스 테스트까지 유지하는 임시 대칭 규칙이다.

## 13. 관측·추론 로그

디버그·테스트 로그는 다음을 남길 수 있다.

- 프로필 키
- 허용된 관측 항목의 종류와 공개 값
- 행동 후보와 점수
- 선택 행동과 사유 코드
- 사용한 결정적 시드 또는 순번

다음은 로그에 남기지 않는다.

- 플레이어 비공개 실제 카드
- 덱의 다음 카드
- 정답과 비교한 내부 치트 값

릴리스 로그 수준과 저장 정책은 별도 결정 전까지 테스트·디버그 범위로 제한한다.

## 16. 성능·품질 기준

- 일반 정책은 한 행동 결정에서 유한한 후보 목록만 평가한다.
- 전투 객체 전체 복제나 모든 덱 순열 탐색을 하지 않는다.
- 관측 모델은 결정 시점 스냅샷이며 외부에서 변경할 수 없다.
- 정책과 카탈로그는 Unity API에 의존하지 않는 순수 C#으로 작성한다.
- 새 외부 패키지·이미지·오디오·폰트를 사용하지 않는다.
- 카드 정의와 기존 테스트 유틸리티를 재사용한다.
