# AI 활용 기술 문서

> 프로젝트: DiaBlackJack  
> 문서 책임자: 이천서  
> 버전: v0.1  
> 최종 갱신: 2026-07-20

## 1. 문서 목적

이 문서는 프로젝트에서 사용한 AI의 역할, 입력 구조, 주요 지시사항, 산출물 검증 방법과 외부 에셋·오픈소스 출처를 기록한다. 제출 및 인수인계 시 사람이 결정한 내용과 AI가 보조한 내용을 구분할 수 있게 하는 것이 목적이다.

이 문서에서 다음 두 종류의 AI를 구분한다.

- **개발 보조 AI**: 기획 정리, 문서 작성, 코드 작성·검토, 테스트 보조에 사용하는 생성형 AI.
- **게임 내 적 AI**: 공개된 게임 상태를 바탕으로 적 행동을 선택하는 규칙 기반 의사결정 시스템.

## 2. 개발 보조 AI 활용 구조

### 2.1 작업 흐름

1. 이천서가 목표, 범위, 기준 문서와 금지 사항을 제시한다.
2. AI가 저장소와 기준 문서를 읽고 초안 또는 변경안을 만든다.
3. AI가 문서 간 충돌, 누락, 파일 상태와 형식을 자체 점검한다.
4. 이천서가 기획 의도와 실제 구현 결과를 검토하고 최종 승인한다.
5. 승인된 결과만 프로젝트 문서와 코드에 반영하며, 사용 내역을 본 문서와 팀 역할 기술서에 요약한다.

개발 보조 AI는 결정권자가 아니다. 게임 규칙, 밸런스, 역할 배정과 외부 자료 사용 승인은 이천서 및 팀이 담당한다.

### 2.2 AI에 제공하는 입력

- `Docs/rule.md`: 게임 규칙의 기준
- `Docs/game-design-document.md`: 전체 기획과 임시 결정
- Unity 프로젝트 구조와 패키지 정보
- 해당 작업의 구현 범위와 제외 범위
- 기존 코드, 테스트, 로그 및 오류 메시지
- 문서 작성자와 실제 기여 기록 원칙

비밀번호, API 키, 개인정보, 공개하면 안 되는 팀 자료는 입력하지 않는다.

### 2.3 주요 프롬프트 및 지시사항

실제 대화의 구어체 표현은 제외하고 기술적 의도만 정리한다.

#### 기획 문서 작성

```text
Docs/rule.md를 기준으로 게임 기획 문서를 작성한다.
규칙에 없는 사항은 사실처럼 단정하지 말고 '임시 기획 결정' 또는
'추후 검증 항목'으로 분리한다. 3인, 2주 프로토타입 범위와 실제 구현
가능성을 고려하며 결과를 Docs에 저장한다.
```

#### 코어 루프 문서화

```text
실제 코드는 작성하지 않는다. 게임의 최소 코어 루프만 정의한다.
런 시작, 전투 시작, 라운드 배분, 히트/스탠드, 승패 판정, 영혼 피해,
전투 종료와 재시작을 포함한다. 카드 특수 효과, 악마 계약, 덱 빌딩,
고급 AI와 연출은 제외한다. 기획 문서와 개발 명세를 분리한다.
```

#### 코드 구현 시 사용할 공통 지시

```text
기준 문서의 상태 전이와 완료 조건을 먼저 확인한다.
기존 Unity 구조와 유틸리티를 재사용하고 새 의존성을 추가하지 않는다.
게임 규칙과 화면 표시를 분리하고, 무작위 처리는 재현 가능한 RNG를
사용한다. 변경 후 EditMode 테스트와 실제 씬 흐름을 검증한다.
```

```text
SP-01 범위에서는 Unity 씬과 전투 객체에 의존하지 않는 순수 C# 상태 기반만 구현한다.
스테이지 정의와 순서, 플레이어의 런 지속 영혼·고정 덱, 시작·클리어·다음 스테이지·

상태를 변경하지 않게 하고 EditMode 단위 테스트와 전체 회귀 테스트로 확인한다.
```

#### 런·스테이지 SP-02 전투 연결 구현

```text
기존 코어 루프의 전투 규칙과 단일 전투 재시작 의미를 바꾸지 않는다. 최대 영혼으로
시작하는 기존 생성자를 보존하고 현재 영혼을 전달하는 호환 경로만 추가한다. 진행 세션은
전투 행동을 기존 CoreLoopSession에 전달하고, 종료된 전투의 결과와 남은 플레이어 영혼을
RunProgress에 한 번만 반영한다. 다음 스테이지에서는 유지된 덱 구성으로 플레이어 덱을
새로 만들고 적 상태를 초기화한다. UI와 씬 전환은 SP-03까지 구현하지 않는다.
```

#### 런·스테이지 SP-03 진행 화면과 씬 연결 구현

```text
SP-03 범위에서 진행 표시 모델, View, Controller와 씬 간 런 세션 유지 계층을
구현한다. 이천서가 미리 만든 StageTest를 진행 전용 씬으로 사용하고 기존
CoreLoopTest를 모든 스테이지의 공용 전투 씬으로 재사용한다. 스테이지별 전투
씬은 만들지 않는다. 진행 상태에 맞는 입력 하나만 활성화하고 View는 규칙을
계산하지 않는다. 기존 CoreLoopTest 단독 실행 동작을 보존하며 표시 단위 테스트,
씬 검증, 런 시작·스테이지 완료·다음 스테이지 진입 흐름으로 확인한다.
```

```text
10회 반복해 상태 초기화, 이벤트 구독과 진행 불가 여부를 점검한다. Test Framework
기반 시설 메시지와 게임 오류를 구분하고 씬·빌드·Console 결과, 이천서의 실제
담당과 AI 검증 보조 내용을 문서에 정제해 기록한다.
```

```text
완료된 전투 시스템의 추가 코어 루프 단계로 부르지 않고 전투 행동 확장이라는
별도 작업으로 관리한다. rule.md의 상세 행동 규칙과 전체 기획서를 대조해 폴드와

명세서, 구현 계획서와 진행 기록을 작성하고 이천서를 문서·결정 책임자로 기록한다.
```

#### 전투 행동 확장 BA-01 기반 구현

```text
BA-01 범위에서 완성된 폴드·체인지 행동과 UI는 구현하지 않는다. 현재 전투 구조를
유지하면서 체인지 후보 선택 상태, 비공개 카드의 원자적 인출, 덱·버린 더미의 후보
확보 가능 여부와 기존 비공개 카드·후보 2장의 단일 선택 결과만 순수 C#으로 추가한다.
범용 행동 명령이나 롤백 프레임워크를 만들지 않고 잘못된 입력이 상태와 카드 위치를
바꾸지 않는지 EditMode 테스트로 고정한다. 구현 전 전체 기준선을 확인하고 Unity
컴파일, CoreLoop 대상 테스트와 전체 회귀 결과를 단계 기록에 남긴다.
```

#### 전투 행동 확장 BA-02 폴드 구현

```text
BA-02 범위에서 플레이어 폴드만 구현한다. PlayerFold 결과는 플레이어 영혼 피해 1,

구현 전·후 전체 EditMode와 대상 테스트, Unity 컴파일·Console을 검증한다.
```

#### 전투 행동 확장 BA-03 체인지 구현

```text
BA-03 범위에서 체인지 시작과 후보 선택 완료를 기존 명시적 행동 API에 추가한다.
플레이어 차례, 라운드 내 미사용, 비공개 카드 1장과 후보 2장 확보 조건을 모두
검사한 뒤 기존 비공개 카드를 보류하고 후보만 공개한다. 유효한 후보 선택 전에는
일반 행동과 취소를 허용하지 않으며, 잘못된 번호는 카드와 상태를 바꾸지 않는다.
선택 카드만 새 비공개 손패로 넣고 나머지 2장은 버린 더미로 이동한 뒤 적 차례를
기존 경로로 실행한다. 같은 라운드 재사용 거부와 새 라운드 초기화, 카드 총량 보존,
CoreLoopSession 전달을 EditMode 테스트로 검증한다. UI와 런 진행 연결은 확장하지 않는다.
```

#### 전투 행동 확장 BA-04 화면·입력 구현

```text
BA-04 범위에서 기존 CoreLoopTest의 IMGUI 구조를 유지한 채 폴드와 체인지를 실제로
조작할 수 있게 한다. 전투 모델은 가능 여부를 제공하고 Presenter는 비용·위험·후보를
표시 모델로 변환하며 View는 렌더링과 입력 이벤트만 담당한다. 평상시에는 히트·스탠드와
폴드·체인지를 2행으로 표시하고, 체인지 선택 중에는 일반 행동 대신 후보 2개만 노출한다.
영혼 1에서 폴드하면 패배한다는 경고와 라운드당 체인지 사용 완료 상태를 버튼 문구에
표시한다. Controller는 단독 전투와 현재 런 전투에 행동을 전달하되, 런 진행의 종료·영혼
동기화 책임은 BA-05의 StageProgressionSession 연결로 남긴다. 새 씬·패키지를 추가하지
않고 표시 모델·Controller 테스트, 실제 Game View, 씬 검증과 Console로 확인한다.
```

#### 전투 행동 확장 BA-05 런 연동·검증 마감

```text
BA-05 범위에서 StageProgressionSession에 폴드, 체인지 시작과 후보 선택 전달을 추가하고
모든 승인된 행동이 기존 전투 종료 동기화 경로를 지나게 한다. Controller가 런 전투의
CoreLoopBattle을 직접 호출하지 않게 하며, 비치명 폴드는 InBattle을 유지하고 영혼 0
폴드는 RunDefeat와 지속 영혼 0으로 동기화한다. 체인지가 적 버스트를 유발해 전투가
끝나는 경우도 StageCleared 또는 최종 결과로 동기화한다. 시작 전·종료 후 입력 거부,
단독 전투 회귀와 런 전투·재시작을 자동 테스트와 실제 StageTest→CoreLoopTest 흐름으로
확인한다. 새 씬·패키지·외부 에셋을 추가하지 않고 AI 활용과 이천서 담당 기록을 마감한다.
```

#### 카드 사용 시스템 CU-00 문서화

전투 보상 RW-01에서는 AI가 카드 정의 카탈로그와 런 덱 구조를 재사용해 일반·높은 등급 풀, 불변 옵션·제안과 시드 기반 결정적 생성 코드 초안을 작성했다. 후보 단계에는 런 카드 ID를 만들지 않고, 선택 승인을 위한 내부 `PlayerRunState.AddRewardCard`에서만 현재 최대 ID 다음 값을 발급하도록 경계를 분리했다. 최초 덱과 현재 덱을 별도로 보존해 재시작 시 영혼·덱·다음 ID가 함께 복구되며, 같은 정의 키를 여러 번 획득해도 물리 ID가 달라지는 것을 검증했다. 신규 8/8, StageProgression 42/42와 전체 EditMode 164/164가 Unity 6000.3.10f1 배치 실행에서 통과했다. 현재 Codex 세션에 Unity MCP 도구가 노출되지 않아 배치 경로를 사용했으며, 로그의 라이선스 토큰 갱신 메시지는 테스트 종료 코드 0과 결과에 영향을 주지 않았다. 진행 상태·세션·UI·씬·패키지·외부 에셋은 변경하지 않았고 최종 코드 승인 책임은 이천서에게 있다.

전투 보상 RW-02에서는 AI가 공개 즉시 완료 API를 보상 시작·선택·건너뛰기 API로 교체하고, 보류 제안과 해결 결과가 완료 목적지를 소유하도록 상태 코드와 테스트 초안을 작성했다. 사전 조건과 옵션 확인을 덱 변경보다 먼저 수행해 실패 입력의 원자성을 보존했으며, 보스는 높은 등급 제안과 `RunVictory` 목적지만 승인하게 했다. 실제 전투 결과에서 제안을 생성하는 세션 책임은 RW-03이므로 `StageProgressionSession`에만 이름이 명시된 내부 무보상 호환 경계를 임시 유지했고, 표시 계층에는 예외 방지를 위한 상태 문구만 추가했다. Codex 도구 목록에 Unity MCP 함수가 직접 노출되지 않았지만 프로젝트의 기존 MCP for Unity 로컬 HTTP 연결을 사용해 갱신·컴파일·테스트를 수행했다. 신규 7/7, StageProgression 49/49와 전체 EditMode 171/171이 통과했고 Console 컴파일 오류는 0개였다. 씬·패키지·외부 에셋은 변경하지 않았으며 최종 코드 승인 책임은 이천서에게 있다.

전투 보상 RW-03에서는 AI가 `StageProgressionSession`에 보상 생성기와 등급 선택 경계를 주입하고 실제 전투 승리의 영혼 동기화 직후 제안을 한 번만 만들도록 코드와 통합 테스트 초안을 작성했다. 최종 보스는 항상 높은 등급과 `RunVictory`, 일반전 기본값은 일반 등급과 `StageCleared`를 사용하며, 엘리트는 아직 없는 스테이지 종류를 선행 추가하지 않고 등급 선택 함수 주입으로 검증했다. RW-02의 내부 무보상 우회 경로는 제거하고 기존 세션 테스트를 실제 보상 선택·건너뛰기 흐름으로 옮겼다. 신규 테스트 클래스만 지정한 최초 실행은 Unity 테스트 트리 반영 시점 때문에 0개를 찾았으나, 같은 어셈블리 55/55와 전체 EditMode 177/177 실행에서 신규 6개를 포함해 모두 통과했다. 직접 Unity MCP 함수가 노출되지 않은 환경에서는 프로젝트의 기존 로컬 HTTP MCP 세션을 재사용했으며 C# 컴파일 오류는 0개였다. Console의 MCP WebSocket 경고 1건과 Test Framework 사전·사후 처리·결과 저장 메시지 6건은 HTTP 연결·테스트 성공과 무관한 기반 시설 출력으로 분리했다. 진행 화면에서 후보를 선택하는 입력은 RW-04로 남겼고 씬·패키지·외부 에셋은 변경하지 않았다. 최종 코드·화면 승인 책임은 이천서에게 있다.

전투 보상 RW-04에서는 AI가 기존 `StageProgression` 표시·View·Controller 경계를 유지해 보류 제안을 읽기 전용 후보 모델로 변환하고, 후보 3장 선택과 건너뛰기를 세션 API에 전달하는 코드 및 표시 테스트 5개 초안을 작성했다. UI가 등급·완료 목적지·덱 장수·최근 처리 결과만 표시하고 상대 비공개 정보를 포함하지 않도록 고정했다. 첫 컴파일에서 테스트 정리 코드의 `Object`가 `System.Object`와 `UnityEngine.Object` 사이에서 모호한 오류 2건을 확인해 후자로 명시했으며, 전체 에셋 갱신 뒤 신규 5/5, StageProgression 60/60과 전체 EditMode 182/182를 통과했다. 실제 Game View에서 일반 선택·건너뛰기, 다음 스테이지, 보스 높은 등급 보상과 건너뛰기 뒤 `RunVictory`를 확인했고 양쪽 씬 문제는 0개였다. 기존 로컬 HTTP MCP를 재사용했으며 별도 씬·프리팹·패키지·외부 에셋은 추가하지 않았다. 최종 코드·화면 승인 책임은 이천서에게 있다.

전투 보상 RW-05에서는 AI가 완료된 공개 API의 반복 안정성을 고정하는 검증 테스트 5개와 기록 초안을 작성했다. 일반 선택·건너뛰기, 보스 선택·건너뛰기, 패배·재시작을 각각 10회 반복하면서 옵션·정의·제안·런 카드 ID 고유성, 선택 카드의 다음 전투 전달, 건너뛰기 덱 유지, 보스 등급·목적지와 새 런 초기화를 검사했다. 신규 5/5, CoreLoop 122/122, StageProgression 65/65, 카드 사용 검증 5/5와 전체 EditMode 187/187이 실패·건너뜀 없이 통과했다. 직접 노출된 Unity 도구가 없는 환경에서 기존 로컬 HTTP MCP를 사용했으며, 검증 코드 실행기의 Roslyn 미지원은 CodeDom 경로로 바꿨다. 최초 실제 흐름용 20장 순서는 보상 상태에 도달하지 않아 프로젝트 데이터를 바꾸지 않고 검증 전용 4장 결정적 덱으로 교정했다. 실제 일반·보스·패배·재시작, 두 씬 문제 0, 시각 판정 98/100과 최종 Console Error/Warning 0을 확인했다. 런타임·씬·프리팹·패키지·외부 에셋은 변경하지 않았으며, 결과 해석과 최종 승인 책임은 이천서에게 있다.

## 3. 게임 내 적 AI 기술 구조

### 3.1 코어 루프 단계의 임시 AI

최소 코어 루프에서는 규칙 검증을 위해 단순 정책만 사용한다.

- 현재 합계가 16 이하면 히트한다.
- 현재 합계가 17 이상이면 스탠드한다.
- 비공개 카드와 자기 카드처럼 정상적으로 접근 가능한 정보만 사용한다.
- 카드 특수 효과, 계약, 플레이어 패 추론은 사용하지 않는다.

2단계 구현의 `SimpleEnemyPolicy`는 적 자신의 `HandValue`만 입력받는다. 플레이어 손패 객체나 덱 순서는 정책 API에 전달하지 않아 비공개 정보가 행동 결정에 섞이지 않도록 했다.

이 정책은 최종 적 AI가 아니라 라운드 흐름을 검증하기 위한 대체 구현이다.

### 3.2 정식 적 AI의 목표 구조

정식 AI는 플레이어 비공개 카드나 덱 순서를 직접 읽지 않는다. 공개 카드, 사용·버린 카드, 공개 덱 구성, 플레이어 행동과 활성 효과를 별도 관측 상태로 전달받아 다음 순서로 행동을 고른다.

1. 공개 정보로 플레이어 비공개 숫자의 확률 분포를 계산한다.
2. 가능한 합계와 버스트 확률을 추정한다.
3. 히트, 스탠드, 폴드, 카드 사용, 계약 후보의 기대 영혼 손익을 비교한다.
4. 적 성향과 난이도 보정을 적용해 행동을 선택한다.
5. 플레이어에게 허용된 범위의 추론 정보만 UI로 공개한다.

### 3.3 공정성과 검증

- AI 입력 모델에서 플레이어 비공개 값과 덱 순서를 제거한다.
- 동일한 공개 상태와 난수 시드는 동일한 결정을 재현해야 한다.
- 디버그 로그에 AI가 참조한 정보와 각 행동 점수를 기록한다.
- UI에 표시된 추론 확률과 내부 확률의 일치 여부를 테스트한다.
- 의도적으로 거짓 정보를 보여 주는 보스는 능력 설명으로 사전 고지한다.

## 4. 생성형 AI 산출물 관리

### 4.1 코드

- 생성 코드는 사람이 구조와 규칙 적합성을 검토한다.
- 프로젝트에 없는 API, 패키지, 클래스 사용 여부를 확인한다.
- 라이선스가 불명확한 외부 코드를 그대로 복사하지 않는다.
- 테스트를 통과하지 않은 코드는 완료로 기록하지 않는다.

### 4.2 이미지·오디오·문구

- 사용한 생성 도구, 생성 날짜, 제작 담당자, 주요 프롬프트를 기록한다.
- 특정 작가나 상표의 스타일 모방을 지시하지 않는다.
- 최종 리소스는 팀이 색상, 선, 광원, 음량과 세계관 적합성을 보정한다.
- 사용 조건과 배포 권리를 확인할 수 없는 결과물은 출시 빌드에 포함하지 않는다.

현재 저장소에서 생성형 이미지·오디오를 최종 게임 에셋으로 사용했다는 기록은 확인되지 않았다.
