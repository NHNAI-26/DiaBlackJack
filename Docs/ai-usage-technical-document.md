# AI 활용 기술 문서

> 프로젝트: DiaBlackJack  
> 문서 책임자: 이천서  
> 버전: v0.1  
> 최종 갱신: 2026-07-20

## 1. 문서 목적

이 문서는 프로젝트에서 사용한 AI의 역할, 입력 구조, 주요 지시사항, 산출물 검증 방법과 외부 에셋·오픈소스 출처를 기록한다. 제출 및 인수인계 시 사람이 결정한 내용과 AI가 보조한 내용을 구분할 수 있게 하는 것이 목적이다.

이 문서에서 다음 두 종류의 AI를 구분한다.

- **개발 보조 AI**: 기획 정리, 문서 작성, 코드 작성·검토, 테스트 보조에 사용하는 생성형 AI.
- **게임 내 적 AI**: 공개된 게임 상태를 바탕으로 적 행동을 선택하는 규칙 기반 의사결정 시스템.

## 2. 개발 보조 AI 활용 구조

### 2.1 작업 흐름

1. 이천서가 목표, 범위, 기준 문서와 금지 사항을 제시한다.
2. AI가 저장소와 기준 문서를 읽고 초안 또는 변경안을 만든다.
3. AI가 문서 간 충돌, 누락, 파일 상태와 형식을 자체 점검한다.
4. 이천서가 기획 의도와 실제 구현 결과를 검토하고 최종 승인한다.
5. 승인된 결과만 프로젝트 문서와 코드에 반영하며, 사용 내역을 본 문서와 팀 역할 기술서에 요약한다.

개발 보조 AI는 결정권자가 아니다. 게임 규칙, 밸런스, 역할 배정과 외부 자료 사용 승인은 이천서 및 팀이 담당한다.

### 2.2 AI에 제공하는 입력

- `Docs/rule.md`: 게임 규칙의 기준
- `Docs/game-design-document.md`: 전체 기획과 임시 결정
- Unity 프로젝트 구조와 패키지 정보
- 해당 작업의 구현 범위와 제외 범위
- 기존 코드, 테스트, 로그 및 오류 메시지
- 문서 작성자와 실제 기여 기록 원칙

비밀번호, API 키, 개인정보, 공개하면 안 되는 팀 자료는 입력하지 않는다.

### 2.3 주요 프롬프트 및 지시사항

실제 대화의 구어체 표현은 제외하고 기술적 의도만 정리한다.

#### 기획 문서 작성

```text
Docs/rule.md를 기준으로 게임 기획 문서를 작성한다.
규칙에 없는 사항은 사실처럼 단정하지 말고 '임시 기획 결정' 또는
'추후 검증 항목'으로 분리한다. 3인, 2주 프로토타입 범위와 실제 구현
가능성을 고려하며 결과를 Docs에 저장한다.
```

#### 코어 루프 문서화

```text
실제 코드는 작성하지 않는다. 게임의 최소 코어 루프만 정의한다.
런 시작, 전투 시작, 라운드 배분, 히트/스탠드, 승패 판정, 영혼 피해,
전투 종료와 재시작을 포함한다. 카드 특수 효과, 악마 계약, 덱 빌딩,
고급 AI와 연출은 제외한다. 기획 문서와 개발 명세를 분리한다.
```

#### 코드 구현 시 사용할 공통 지시

```text
기준 문서의 상태 전이와 완료 조건을 먼저 확인한다.
기존 Unity 구조와 유틸리티를 재사용하고 새 의존성을 추가하지 않는다.
게임 규칙과 화면 표시를 분리하고, 무작위 처리는 재현 가능한 RNG를
사용한다. 변경 후 EditMode 테스트와 실제 씬 흐름을 검증한다.
```

```text
SP-01 범위에서는 Unity 씬과 전투 객체에 의존하지 않는 순수 C# 상태 기반만 구현한다.
스테이지 정의와 순서, 플레이어의 런 지속 영혼·고정 덱, 시작·클리어·다음 스테이지·

상태를 변경하지 않게 하고 EditMode 단위 테스트와 전체 회귀 테스트로 확인한다.
```

#### 런·스테이지 SP-02 전투 연결 구현

```text
기존 코어 루프의 전투 규칙과 단일 전투 재시작 의미를 바꾸지 않는다. 최대 영혼으로
시작하는 기존 생성자를 보존하고 현재 영혼을 전달하는 호환 경로만 추가한다. 진행 세션은
전투 행동을 기존 CoreLoopSession에 전달하고, 종료된 전투의 결과와 남은 플레이어 영혼을
RunProgress에 한 번만 반영한다. 다음 스테이지에서는 유지된 덱 구성으로 플레이어 덱을
새로 만들고 적 상태를 초기화한다. UI와 씬 전환은 SP-03까지 구현하지 않는다.
```

#### 런·스테이지 SP-03 진행 화면과 씬 연결 구현

```text
SP-03 범위에서 진행 표시 모델, View, Controller와 씬 간 런 세션 유지 계층을
구현한다. 이천서가 미리 만든 StageTest를 진행 전용 씬으로 사용하고 기존
CoreLoopTest를 모든 스테이지의 공용 전투 씬으로 재사용한다. 스테이지별 전투
씬은 만들지 않는다. 진행 상태에 맞는 입력 하나만 활성화하고 View는 규칙을
계산하지 않는다. 기존 CoreLoopTest 단독 실행 동작을 보존하며 표시 단위 테스트,
씬 검증, 런 시작·스테이지 완료·다음 스테이지 진입 흐름으로 확인한다.
```

```text
10회 반복해 상태 초기화, 이벤트 구독과 진행 불가 여부를 점검한다. Test Framework
기반 시설 메시지와 게임 오류를 구분하고 씬·빌드·Console 결과, 이천서의 실제
담당과 AI 검증 보조 내용을 문서에 정제해 기록한다.
```

```text
완료된 전투 시스템의 추가 코어 루프 단계로 부르지 않고 전투 행동 확장이라는
별도 작업으로 관리한다. rule.md의 상세 행동 규칙과 전체 기획서를 대조해 폴드와

명세서, 구현 계획서와 진행 기록을 작성하고 이천서를 문서·결정 책임자로 기록한다.
```

#### 전투 행동 확장 BA-01 기반 구현

```text
BA-01 범위에서 완성된 폴드·체인지 행동과 UI는 구현하지 않는다. 현재 전투 구조를
유지하면서 체인지 후보 선택 상태, 비공개 카드의 원자적 인출, 덱·버린 더미의 후보
확보 가능 여부와 기존 비공개 카드·후보 2장의 단일 선택 결과만 순수 C#으로 추가한다.
범용 행동 명령이나 롤백 프레임워크를 만들지 않고 잘못된 입력이 상태와 카드 위치를
바꾸지 않는지 EditMode 테스트로 고정한다. 구현 전 전체 기준선을 확인하고 Unity
컴파일, CoreLoop 대상 테스트와 전체 회귀 결과를 단계 기록에 남긴다.
```

#### 전투 행동 확장 BA-02 폴드 구현

```text
BA-02 범위에서 플레이어 폴드만 구현한다. PlayerFold 결과는 플레이어 영혼 피해 1,

구현 전·후 전체 EditMode와 대상 테스트, Unity 컴파일·Console을 검증한다.
```

#### 전투 행동 확장 BA-03 체인지 구현

```text
BA-03 범위에서 체인지 시작과 후보 선택 완료를 기존 명시적 행동 API에 추가한다.
플레이어 차례, 라운드 내 미사용, 비공개 카드 1장과 후보 2장 확보 조건을 모두
검사한 뒤 기존 비공개 카드를 보류하고 후보만 공개한다. 유효한 후보 선택 전에는
일반 행동과 취소를 허용하지 않으며, 잘못된 번호는 카드와 상태를 바꾸지 않는다.
선택 카드만 새 비공개 손패로 넣고 나머지 2장은 버린 더미로 이동한 뒤 적 차례를
기존 경로로 실행한다. 같은 라운드 재사용 거부와 새 라운드 초기화, 카드 총량 보존,
CoreLoopSession 전달을 EditMode 테스트로 검증한다. UI와 런 진행 연결은 확장하지 않는다.
```

#### 전투 행동 확장 BA-04 화면·입력 구현

```text
BA-04 범위에서 기존 CoreLoopTest의 IMGUI 구조를 유지한 채 폴드와 체인지를 실제로
조작할 수 있게 한다. 전투 모델은 가능 여부를 제공하고 Presenter는 비용·위험·후보를
표시 모델로 변환하며 View는 렌더링과 입력 이벤트만 담당한다. 평상시에는 히트·스탠드와
폴드·체인지를 2행으로 표시하고, 체인지 선택 중에는 일반 행동 대신 후보 2개만 노출한다.
영혼 1에서 폴드하면 패배한다는 경고와 라운드당 체인지 사용 완료 상태를 버튼 문구에
표시한다. Controller는 단독 전투와 현재 런 전투에 행동을 전달하되, 런 진행의 종료·영혼
동기화 책임은 BA-05의 StageProgressionSession 연결로 남긴다. 새 씬·패키지를 추가하지
않고 표시 모델·Controller 테스트, 실제 Game View, 씬 검증과 Console로 확인한다.
```

#### 전투 행동 확장 BA-05 런 연동·검증 마감

```text
BA-05 범위에서 StageProgressionSession에 폴드, 체인지 시작과 후보 선택 전달을 추가하고
모든 승인된 행동이 기존 전투 종료 동기화 경로를 지나게 한다. Controller가 런 전투의
CoreLoopBattle을 직접 호출하지 않게 하며, 비치명 폴드는 InBattle을 유지하고 영혼 0
폴드는 RunDefeat와 지속 영혼 0으로 동기화한다. 체인지가 적 버스트를 유발해 전투가
끝나는 경우도 StageCleared 또는 최종 결과로 동기화한다. 시작 전·종료 후 입력 거부,
단독 전투 회귀와 런 전투·재시작을 자동 테스트와 실제 StageTest→CoreLoopTest 흐름으로
확인한다. 새 씬·패키지·외부 에셋을 추가하지 않고 AI 활용과 이천서 담당 기록을 마감한다.
```

#### 카드 사용 시스템 CU-00 문서화

상대 선택·적 전투 정보 UI EUI-02에서는 AI가 후보 표시 모델, 세션 기반 Presenter 오버로드, 기존 IMGUI 후보 카드와 Controller 로컬 집중·씬 이동 분기, 프로토타입 Runtime 선택 기능 활성화 초안을 작성하고 Unity MCP 검증을 보조했다. 후보 표시값은 `EnemyProfilePreview`의 공개 필드만 매핑하고 현재 Offer에 속한 정확한 키 하나만 집중·확정 가능하게 만들었다. 시작 입력은 `OpponentSelection` 상태에서 `StageTest`를 유지했으며 실제 Play Mode에서 후보 2명, 집중 강조, 확정 버튼 활성 상태를 확인했다. 신규 9/9, StageProgression 103/103, CoreLoop 179/179, 전체 EditMode 282/282와 변경 5개 스크립트 진단 0, 1280×720·1920×1080 화면에서 겹침·잘림 없음을 확인했다. 최초 Runtime 직접 호출 EditMode 테스트는 `DontDestroyOnLoad`의 Play Mode 제한 때문에 제거하고 실제 Play Mode로 대체했다. 실제 확정 API·전투 생성·전투 씬 이동은 EUI-03으로 남겼으며 씬·프리팹·Packages·외부 에셋은 변경하지 않았다. 최종 코드·화면 승인 책임은 이천서에게 있다.

상대 선택·적 전투 정보 UI EUI-03에서는 AI가 현재 제안의 `OfferId+ProfileKey`만 허용하는 원자적 세션 확정 API, 기존 View 확정 이벤트의 Controller 전달과 통합 테스트 초안을 작성하고 Unity MCP 검증을 보조했다. 세션은 입력과 현재 스테이지를 먼저 검증하고 선택 미리보기와 스테이지 템플릿·덱 시드를 결합한 뒤 EP-06 Factory로 실제 전투를 생성하며, 생성과 진행 전이가 모두 성공한 뒤에만 Pending Offer·ActiveStage·Battle을 교체한다. 네 선택 프로필의 최대 영혼·10장 덱·전용 정책, 잘못된 입력과 Factory 예외의 무변경, 일반/엘리트 보상, 두 번의 선택·고정 보스·보스 승리 뒤 새 런 초기화를 14개 통합 테스트로 고정했다. 최초 컴파일의 CS0051은 public NUnit 메서드가 private 열거형을 매개변수로 사용한 문제여서 열거형 접근 수준을 맞췄다. 최종 신규 14/14, StageProgression 117/117, CoreLoop 179/179, 전체 EditMode 296/296과 변경 3개 C# 정적 진단 0을 확인했다. 실제 Play Mode에서 집행관 선택이 최대 영혼 5·적 덱 10장의 전투를 만들고 다음 프레임 `CoreLoopTest`와 전투 ViewModel로 이어짐을 확인했다. 씬·프리팹·Packages·외부 에셋은 변경하지 않았으며 최종 코드 승인 책임은 이천서에게 있다.

## 3. 게임 내 적 AI 기술 구조

### 3.1 코어 루프 단계의 임시 AI

최소 코어 루프에서는 규칙 검증을 위해 단순 정책만 사용한다.

- 현재 합계가 16 이하면 히트한다.
- 현재 합계가 17 이상이면 스탠드한다.
- 비공개 카드와 자기 카드처럼 정상적으로 접근 가능한 정보만 사용한다.
- 카드 특수 효과, 계약, 플레이어 패 추론은 사용하지 않는다.

2단계 구현의 `SimpleEnemyPolicy`는 적 자신의 `HandValue`만 입력받는다. 플레이어 손패 객체나 덱 순서는 정책 API에 전달하지 않아 비공개 정보가 행동 결정에 섞이지 않도록 했다.

이 정책은 최종 적 AI가 아니라 라운드 흐름을 검증하기 위한 대체 구현이다.

### 3.2 정식 적 AI의 목표 구조

정식 AI는 플레이어 비공개 카드나 덱 순서를 직접 읽지 않는다. 공개 카드, 사용·버린 카드, 공개 덱 구성, 플레이어 행동과 활성 효과를 별도 관측 상태로 전달받아 다음 순서로 행동을 고른다.

1. 공개 정보로 플레이어 비공개 숫자의 확률 분포를 계산한다.
2. 가능한 합계와 버스트 확률을 추정한다.
3. 히트, 스탠드, 폴드, 카드 사용, 계약 후보의 기대 영혼 손익을 비교한다.
4. 적 성향과 난이도 보정을 적용해 행동을 선택한다.
5. 플레이어에게 허용된 범위의 추론 정보만 UI로 공개한다.

### 3.3 공정성과 검증

- AI 입력 모델에서 플레이어 비공개 값과 덱 순서를 제거한다.
- 동일한 공개 상태와 난수 시드는 동일한 결정을 재현해야 한다.
- 디버그 로그에 AI가 참조한 정보와 각 행동 점수를 기록한다.
- UI에 표시된 추론 확률과 내부 확률의 일치 여부를 테스트한다.
- 의도적으로 거짓 정보를 보여 주는 보스는 능력 설명으로 사전 고지한다.

### 3.4 EP 목표 구조와 담당

EP 작업에서는 기존 `SimpleEnemyPolicy`를 호환 기본값으로 유지하면서 공개 관측 스냅샷을 받는 교체 가능 행동 정책으로 확장한다. 안정적인 전투 프로필 키가 적 등급·최대 영혼·덱·정책과 표시 미리보기의 단일 기준이 되며, 상대 선택 시스템은 이 키만 전달한다.

- 이천서: 적 프로필·관측·행동 정책·적 카드 효과·실제 전투 생성과 보상 연결
- 개발 팀원: 상대 후보 생성·선택 상태·선택 UI, 프로필 키 소비
- 아트 팀원: 적 모델·초상·애니메이션·전투 공간과 효과 리소스

세부 타입·단계·검증 기준은 `enemy-combat-profile-development-spec.md`와 `enemy-combat-profile-implementation-plan.md`를 따른다.

### 3.5 EP-02 구현 구조

EP-02부터 `EnemyObservationFactory`가 전투 객체를 정책용 불변 공개 스냅샷으로 변환하고, `EnemyActionCandidate`가 그 시점에 실행 가능한 선택지만 제공한다. 정책은 후보별 점수와 사유를 가진 `EnemyDecision`을 반환하며, `EnemyDecisionValidator`가 카드 ID와 효과 옵션까지 실행 직전에 다시 확인한다. 카드 효과는 `CombatantSide`로 행위자와 상대를 결정하므로 같은 처리기가 플레이어와 적 모두에게 적용된다. 공개 카드 기반 확률 추론과 적별 가중치는 EP-03 이후 정책에서 추가한다.

### 3.6 EP-03 일반 적 정책 구조

`EnemyNumberInferenceCalculator`는 플레이어 덱의 숫자별 전체 장수와 공개된 카드만 받아 비공개 후보 확률을 만들며, 덱의 저장 순서나 비공개 카드 객체를 받지 않는다. `EnemyPolicyDecisionSelector`는 세 정책이 같은 유효 후보·점수 기록 구조를 사용하게 하고, `GunslingerEnemyPolicy`, `CultistEnemyPolicy`, `TricksterEnemyPolicy`가 성향별 점수만 제공한다. 자동 권총 숫자와 수정 구슬 임시 카드는 적이 규칙상 확인할 수 있는 선택 옵션 정보로 후보에 포함된다. 집행관과 보스는 각 후속 단계 전까지 `SimpleEnemyPolicy`를 유지한다.

### 3.7 EP-04 집행관 엘리트 정책 구조

`EnforcerEnemyPolicy`는 같은 공개 관측과 유효 후보 경계를 사용하되 해머·나이프 뒤 상태를 한 단계 더 평가한다. 해머 선택 옵션은 `EnemyObservationFactory`가 효과 임시 카드 또는 적 손패의 실제 소유 공개 카드에서만 숫자를 복사하며, 정책은 최저 숫자 비용을 고른다. 나이프의 버스트 가능성은 공개 손패와 `EnemyNumberInference` 분포를 조합한 결정적 근사값이며 실제 비공개 카드 객체나 덱 순서를 사용하지 않는다. `EnemyInferenceDisplayModel`은 원시 확률 대신 상위 숫자 2개와 신뢰도 열거값만 외부에 제공한다. 보상은 새 시스템을 만들지 않고 프로필의 `ExpectedRewardTier`와 기존 `StageProgressionSession` 완료 목적지를 재사용한다.

### 3.8 EP-05 최종 보스 3구간 정책 구조

`FinalBossPhaseResolver`는 최대 영혼 7의 현재 영혼만 받아 생존·압박·처형 구간을 결정한다. `FinalBossEnemyPolicy`는 기존 `EnemyPolicyDecisionSelector`와 카드 선택 후보를 재사용하며 처형 구간의 예고 범주, 라운드 번호와 공개 플레이어 행동 수만 내부 상태로 보존한다. 같은 라운드에서 플레이어 행동 수가 증가하고 해당 카드 후보가 여전히 유효할 때만 예고 행동을 실행한다. `BossCombatDisplayModel`은 구간, 추론 방향, 신뢰도와 예고 범주 네 속성만 외부에 제공하므로 실제 비공개 숫자나 카드 식별자로 역추론할 수 없다. 전투 종료·보상·재시작은 별도 보스 상태 기계 없이 기존 `CoreLoopBattle`과 `StageProgressionSession`을 그대로 사용한다.

### 3.9 EP-06 선택 전투 변환 구조

`StageDefinition.CreateForEnemyProfile(...)`는 안정적인 키로 `EnemyProfilePreview`를 조회해 최대 영혼을 파생하고 보스 등급과 스테이지 종류를 교차 검증한다. `StageBattleFactory`는 저장된 키를 `EnemyBattleConfigurationFactory`에 전달해 플레이어 런 덱과 전용 적 덱, 최대 영혼과 새 행동 정책으로 전투를 만든다. `StageProgressionSession`은 같은 키의 예상 보상 등급을 기본값으로 사용하고 최종 보스의 보상 완료 목적지는 기존 `RunVictory` 경계를 유지한다. 이 구조에서 상대 선택 담당은 정책 객체나 영혼·보상 값을 만들지 않으며, 키 없는 과거 스테이지는 표준 덱·기본 정책 호환 경로를 사용한다.

### 3.10 EUI 목표 표시 구조

상대 선택 화면은 `EnemyCombatProfileCatalog`의 안전 미리보기만 후보 모델로 변환하고 선택 확정 시 프로필 키 하나만 진행 세션에 전달한다. 진행 세션은 이 키를 EP-06 변환 경계에 넘겨 실제 덱·영혼·정책·보상을 만든다. 전투 화면은 현재 전투와 선택된 프로필 키를 읽되, 별도의 `EnemyCombatDisplaySnapshot` 생성 경계에서 공개 관측과 기존 등급별 표시 모델만 소비한다. 표시를 만들기 위해 적 정책의 `Decide`를 추가 호출하지 않으므로 예고·정책 상태와 실제 행동 순서가 변하지 않는다. 프로필 키가 없는 독립 전투는 일반 정보만 표시하는 호환 경로를 유지한다.

EUI-01 구현에서는 `OpponentSelectionGenerator`가 카탈로그 미리보기를 일반·엘리트 풀로 한 번 분류하고, 생성기 인스턴스 내부 난수와 OfferId만 가변 상태로 보존한다. `StageProgressionSession`은 생성기 주입 여부로 기존 고정 흐름과 선택 흐름을 분기하며, 선택 중에는 전투 세션을 만들지 않아 기존 행동 전달 경계가 자연스럽게 입력을 거절한다. 최종 보스는 선택 제안을 만들지 않고 원래 `StageDefinition`으로 전투를 생성한다.

EUI-02 구현에서는 `StageProgressionPresenter`가 기존 `RunProgress` 오버로드를 보존하면서 `StageProgressionSession`과 집중 키를 받는 오버로드를 추가했다. ViewModel은 후보 표시 목록과 OfferId·집중 키·집중/확정 가능 여부만 노출한다. `StageProgressionController`는 집중 상태를 세션 밖에서 관리하고 OfferId 변경 시 초기화하며, 진행 입력 성공 뒤 `InBattle`과 실제 Battle이 함께 있을 때만 전투 씬을 연다. 이 경계 때문에 Runtime에서 선택 생성기를 켜도 Battle null 상태의 잘못된 씬 전환이 발생하지 않는다.

EUI-03 구현에서는 `StageProgressionSession.TrySelectOpponent`가 상태·현재 제안·제안 ID·스테이지 번호·정확한 후보 키를 전부 검사한 뒤 선택 미리보기와 현재 템플릿으로 새 `StageDefinition`을 만든다. `StageBattleFactory`와 `CoreLoopSession.StartNewBattle`이 성공하기 전에는 세션 필드를 바꾸지 않고, `RunProgress` 전이까지 성공한 뒤 ActiveStage·Pending Offer·Battle을 한 번에 교체한다. Controller는 이미 집중된 키와 ViewModel의 OfferId만 전달하고 기존 입력 잠금·라우팅을 재사용하므로 성공한 전투만 `CoreLoopTest`로 이동한다. 보상 완료 뒤에는 같은 생성기에서 다음 OfferId를 만들고, 두 일반전 뒤에는 기존 고정 보스로 우회하며, 재시작은 새 생성기로 OfferId 0부터 복구한다.

## 4. 생성형 AI 산출물 관리

### 4.1 코드

- 생성 코드는 사람이 구조와 규칙 적합성을 검토한다.
- 프로젝트에 없는 API, 패키지, 클래스 사용 여부를 확인한다.
- 라이선스가 불명확한 외부 코드를 그대로 복사하지 않는다.
- 테스트를 통과하지 않은 코드는 완료로 기록하지 않는다.

### 4.2 이미지·오디오·문구

- 사용한 생성 도구, 생성 날짜, 제작 담당자, 주요 프롬프트를 기록한다.
- 특정 작가나 상표의 스타일 모방을 지시하지 않는다.
- 최종 리소스는 팀이 색상, 선, 광원, 음량과 세계관 적합성을 보정한다.
- 사용 조건과 배포 권리를 확인할 수 없는 결과물은 출시 빌드에 포함하지 않는다.

현재 저장소에서 생성형 이미지·오디오를 최종 게임 에셋으로 사용했다는 기록은 확인되지 않았다.

## 6. 미확인 사항과 후속 작업

- `Assets/Wingman`의 원 출처와 라이선스를 확인한다.
- 후속 기능을 구현할 때 사용한 AI 모델/도구, 주요 지시, 수정 내역과 테스트 결과를 작업 단위로 계속 추가한다.
- 생성형 이미지나 음원을 사용하면 도구·프롬프트·권리 검토 결과를 별도 자산 대장에 기록한다.
