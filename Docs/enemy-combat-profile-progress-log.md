## 1. 기록 원칙

- 계획과 실제 구현을 구분한다.
- 문서 단계는 코드 구현 완료로 기록하지 않는다.
- 적 AI가 사용한 공개·비공개 정보 경계를 단계마다 기록한다.
- 변경 파일, 테스트, 실제 전투, 씬·패키지·외부 에셋 변경 여부를 남긴다.
- 상대 선택·아트 담당자의 파일을 이천서의 실제 기여로 기록하지 않는다.
- AI 입력 원문은 목적, 지시, 결과와 사람의 결정으로 정제한다.
- 기획·코드·검증의 최종 책임자는 이천서로 기록한다.
- 사용자의 요청 전에는 스테이징·커밋·푸시하지 않는다.

## 3. 착수 기준선

### 3.1 완료된 선행 작업

- 코어 루프 1~4단계 완료
- 런·스테이지 SP-00~SP-04 완료
- 전투 행동 BA-00~BA-05 완료
- 카드 사용 CU-00~CU-06 완료
- 전투 보상 RW-00~RW-05 완료
- RW-05 기준 CoreLoop 122/122, StageProgression 65/65, 전체 EditMode 187/187 통과
- `CoreLoopTest`, `StageTest` 씬 문제 0과 최종 Console Error/Warning 0 확인

### 3.2 EP-00 시점 코드 상태

- `SimpleEnemyPolicy`는 적 자신의 합계만 보고 16 이하 히트, 17 이상 스탠드한다.
- `CoreLoopBattle`은 구체 `SimpleEnemyPolicy`에 의존한다.
- 적 차례의 카드 사용·폴드·체인지·계약 API는 없다.
- 수정 구슬·위협용 해머·자동 권총·군용 나이프의 플레이어 효과 처리기는 존재한다.
- 거짓말 탐지기와 악마 계약은 현재 카드 카탈로그·처리기에 없다.
- `StageBattleFactory`는 적 표준 덱과 스테이지 최대 영혼으로 전투를 만든다.
- `StageProgressionSession`에는 전투 생성과 보상 등급 주입 경계가 있다.
- 상대 후보와 선택된 적 프로필 키는 아직 없다.

### 3.3 협업·Git 주의사항

- `StageProgressionSession.cs`, `StageBattleFactory.cs`, 진행 UI와 테스트 씬은 상대 선택 통합 시 충돌 가능성이 높은 공유 파일이다.
- 공유 파일은 EP-06 전에 담당자를 한 명 정하고 동시 수정하지 않는다.
- 아트 담당 경로와 Unity 자동 생성 솔루션 변경은 EP 문서 커밋에서 제외한다.
- 현재 Unity가 자동 갱신한 `DiaBlackJack.slnx` 변경은 의미 있는 EP 변경으로 포함하지 않는다.

## 5. 단계별 기록

### EP-00 — 구현 기준 확정

#### 수행 내용

- `rule.md`와 전체 게임 기획의 적 AI 공정성, 적 유형, 영혼, 보스 구간과 보상 규칙을 대조했다.
- 현재 `SimpleEnemyPolicy`, `CoreLoopBattle`, 카드 정의·효과 처리기, `StageBattleFactory`와 보상 등급 주입 경계를 확인했다.
- 적 전투 프로필 기획서, 개발 명세서, 구현 계획서와 진행 기록을 작성했다.
- 일반 적 3종, 엘리트와 보스의 키·영혼·행동 성향·카드 의존성을 확정했다.
- 공개 관측, 정책 결정·실행, 프로필 기반 전투 생성과 상대 선택 협업 계약을 명세했다.
- EP-01~EP-06 작업, 테스트, 파일 소유권과 완료 게이트를 작성했다.
- AI 활용 기술서와 이천서 역할 기록을 갱신 대상으로 포함했다.

#### 변경 범위

- 적 전투 프로필 Markdown 문서 4종 신규 작성
- README, AI 활용 기술서와 팀 역할 기술서 갱신
- 게임 런타임 코드 변경 없음
- 테스트 코드와 Unity 테스트 실행 없음
- 씬·프리팹·패키지·외부 에셋 변경 없음
- 다른 팀원의 상대 선택·아트 파일 변경 없음

#### 검증

- 기준 규칙과 적 5종·영혼·행동·보상 대조
- 현재 CoreLoop 정책·카드·전투 생성 API 정적 대조
- 문서 4종의 범위·작업 ID·의존성·완료 게이트 대조
- README 링크, 이천서 이름, AI 활용·역할 기록 확인
- Git diff 공백 오류 확인
- 문서 전용 단계이므로 Unity Editor와 EditMode 테스트는 다시 실행하지 않음

#### 남은 위험

- 적 카드 사용은 현재 플레이어 행위자를 전제로 한 코드가 있는지 EP-02에서 확인해야 한다.
- 광신도의 계약과 사기꾼의 거짓말 탐지기는 구현 의존성이 없어 1차 성향이 제한된다.
- 상대 선택 담당이 사용할 프로필 미리보기 API는 EP-01 구현 뒤 확정된다.
- 보스 행동 가중치와 전투 시간은 실제 플레이 테스트 전까지 임시 값이다.

#### 추천 커밋 제목

`다섯 적의 전투 성향과 상대 선택 연동 경계를 구현 기준으로 확정`

### EP-01 — 프로필·정책 기반

#### 수행 내용

- `EnemyGrade`, `EnemyInformationMode`, `EnemyCombatProfile`과 읽기 전용 `EnemyProfilePreview`를 작성했다.
- 총잡이·광신도·사기꾼·집행관·최종 보스의 안정적인 키, 등급, 최대 영혼, 구현 완료 카드만 사용한 덱과 요약을 기본 카탈로그에 등록했다.
- 프로필 생성 시 빈 값, 잘못된 열거값·영혼·정책·덱·카드 정의를 거절하고 카탈로그의 빈 목록·null·중복 키를 거절했다.
- `IEnemyBehaviorPolicy`, 최소 `EnemyObservation`, `EnemyDecision`과 정책 카탈로그를 작성했다.
- `SimpleEnemyPolicy`의 기존 `Decide(HandValue)` API와 16/17 규칙을 유지하면서 새 정책 인터페이스를 구현했다.
- `CoreLoopBattle`이 구체 정책 대신 인터페이스를 받도록 바꾸고 기존 기본 생성 경로는 단순 정책을 계속 사용하게 했다.
- 프로필 키와 시드로 카드 정의·최대 영혼·정책·예상 보상 등급을 결정적으로 만드는 `EnemyBattleConfigurationFactory`를 작성했다.
- 상대 선택 담당자는 `EnemyCombatProfileCatalog.Default.Previews`와 선택한 `ProfileKey`만 사용하면 되도록 내부 덱·정책을 미리보기에서 제외했다.

#### 변경 범위

- `Assets/01. Scripts/Runtime/CoreLoop/EnemyAI` 순수 C# 정책 기반 신규 작성
- `Assets/01. Scripts/Runtime/CoreLoop/EnemyProfiles` 순수 C# 프로필·카탈로그·전투 설정 신규 작성
- `CoreLoopBattle.cs`, `SimpleEnemyPolicy.cs` 호환 경계 변경
- `EnemyProfileFoundationTests.cs` 신규 테스트 12개 작성
- 기존 `Border` 어셈블리 유지, 새 asmdef·외부 의존성 없음
- 씬·프리팹·패키지·이미지·오디오·폰트 변경 없음
- Unity 자동 생성 `DiaBlackJack.slnx` 변경은 작업 범위에서 제외

#### 검증

- 전체 에셋 강제 갱신 뒤 Unity 컴파일 오류 0
- EP-01 신규 EditMode 12/12 통과
- CoreLoop EditMode 134/134 통과
- 전체 EditMode 199/199 통과
- 잘못된 프로필 키가 기본 적으로 자동 대체되지 않음을 확인
- 같은 프로필·시드의 설정과 실제 덱 드로우 순서가 같음을 확인
- 기존 16 이하 히트·17 이상 스탠드와 교체 정책 주입을 확인

#### 남은 위험

- 최소 `EnemyObservation`은 현재 적 자신의 손패 합계만 보존하며 공개 카드·행동 이력·카드 가용성은 EP-02에서 추가해야 한다.
- `Fold`와 `UseCard` 결정 타입은 정의됐지만 실행기는 EP-02 범위이므로 현재 전투는 히트·스탠드만 승인한다.
- 다섯 프로필의 덱 구성은 구현 가능성 확인용 임시안이며 적별 밸런스와 특수 정책은 EP-03~EP-05에서 검증해야 한다.
- 상대 선택 후보 생성·UI와 실제 `StageBattleFactory` 연결은 다른 담당 및 EP-06 통합 범위다.

## 7. EP-02 착수 조건

1. `EnemyObservation`에 포함할 공개 정보와 금지할 플레이어 비공개 값·덱 순서를 타입 단위로 확정한다.
2. 히트·스탠드·폴드·카드 사용의 승인 전 가용성 모델과 결정 점수 결과를 정의한다.
3. 잘못된 정책 결정이 전투 상태를 바꾸기 전에 거절되는 실행 경계를 설계한다.

5. EP-01 전체 EditMode 199/199을 회귀 기준선으로 사용한다.

EP-02도 UI·씬과 분리된 순수 전투 규칙부터 구현하며, 적별 가중치와 개별 성향은 EP-03 이후에 작성한다.
