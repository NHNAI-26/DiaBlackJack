## 1. 문서 목적

현재 전투 시스템의 히트·스탠드에 **폴드**와 **체인지**를 추가해, 플레이어가 불리한 패를 영혼 비용 또는 카드 교체 선택으로 통제할 수 있게 한다.

이 작업은 완료된 코어 루프의 추가 단계가 아니라 별도의 **전투 행동 확장** 작업으로 관리한다. 기존 전투·런·스테이지 진행 구조를 재작성하지 않고, 명시적인 행동 입력과 상태만 확장한다.

기준 문서는 `Docs/rule.md`와 `Docs/game-design-document.md`다. 두 문서의 충돌이나 구현을 위해 정한 세부 규칙은 이 문서의 임시 기획 결정으로 구분한다.

## 3. 기준 문서 충돌 정리

`rule.md` 7.2와 `game-design-document.md` 4.3은 히트, 스탠드, 폴드, 카드 사용, 체인지, 계약의 6개 행동을 정의한다. 반면 `rule.md`의 프로토타입 필수 전투 목록에는 체인지가 빠져 있다.

이번 작업에서는 다음 근거로 체인지 누락을 범위 제외가 아닌 목록 누락으로 처리한다.

- 전체 기획서의 프로토타입 완료 정의가 핵심 행동 6종을 요구한다.
- 2주 일정에서 폴드·체인지를 같은 구현 묶음으로 배치한다.
- 운에 의한 패배를 줄이는 통제 수단으로 체인지를 명시한다.
- 체인지 상세 규칙이 이미 별도로 정의되어 있다.

따라서 이천서가 관리하는 1차 전투 행동 확장에는 폴드와 체인지를 포함한다. 카드 사용과 계약은 문서상 핵심 행동이지만 효과 처리, 선택 대상, 지속 상태가 필요하므로 별도 작업으로 분리한다.

## 4. 작업 범위

### 4.1 포함 범위

- 플레이어 폴드 입력과 사용 조건
- 폴드 시 플레이어 영혼 1 감소
- 폴드 즉시 라운드 종료와 새 라운드 시작
- 폴드로 영혼이 0이 될 때 전투·런 패배 연결
- 플레이어 체인지 입력과 라운드당 1회 제한
- 기존 비공개 카드 교체
- 덱 위 후보 2장을 플레이어에게만 표시
- 후보 1장 선택, 선택하지 않은 카드 폐기
- 체인지 완료 후 적 차례 진행
- 기존 전투 UI와 런·스테이지 진행 세션 연결
- 자동 테스트와 실제 씬 회귀 검증

### 4.2 제외 범위

- 일반 카드의 사용 효과와 자동 발동 효과
- 카드 대상 선택, 효과 처리 대기열과 연쇄 효과
- 악마 계약 후보, 비용, 지속 효과와 대가
- 적의 폴드·체인지·카드 사용·계약 판단
- 행동 취소·되돌리기와 확인 팝업
- 최종 아트, 애니메이션, 사운드와 튜토리얼
- 행동 이력 기반 추론 AI

제외 기능을 대비한 범용 명령 시스템이나 빈 효과 프레임워크는 이번 작업에서 만들지 않는다.

## 5. 플레이 경험 목표

폴드는 영혼 2를 잃을 가능성이 높은 라운드를 영혼 1 손실로 조기에 끝내는 방어적 선택이다. 체인지는 영혼을 소비하지 않는 대신 라운드당 한 번만 사용할 수 있고, 후보를 확인한 뒤 반드시 하나를 선택해야 하는 확률 통제 수단이다.

두 행동은 다음 판단을 만든다.

- 현재 패로 계속 진행할지, 확정 손실을 감수하고 폴드할지 결정한다.
- 체인지를 지금 사용할지, 이후 더 불리한 상황을 위해 보존할지 결정한다.
- 후보 카드의 숫자와 남은 덱 정보를 바탕으로 새 비공개 카드를 선택한다.

## 6. 폴드 규칙

### 6.1 사용 조건

- 전투가 진행 중이고 플레이어 차례여야 한다.
- 플레이어가 스탠드한 상태가 아니어야 한다.
- 이미 라운드 판정 또는 다른 선택 행동을 처리 중이면 사용할 수 없다.
- 현재 영혼이 1이어도 사용할 수 있다. 이 경우 폴드 피해로 영혼이 0이 되어 전투 및 런에서 패배한다.

### 6.2 처리 순서

1. 폴드 입력을 승인한다.
2. 현재 라운드 결과를 `PlayerFold`로 확정한다.
3. 플레이어 영혼에 피해 1을 정확히 한 번 적용한다.
4. 양쪽의 현재 라운드 카드를 각자 버린 카드 더미로 이동한다.
5. 영혼이 0이면 전투 패배로 끝낸다.
6. 전투가 계속되면 남은 덱과 버린 카드 더미를 유지한 채 새 라운드를 시작한다.

폴드는 적 차례를 실행하지 않고 라운드를 즉시 종료한다. 폴드로 끝난 라운드에는 합계 비교, 버스트 피해 또는 정확히 21 피해를 추가로 적용하지 않는다.

## 7. 체인지 규칙

### 7.1 사용 조건

- 전투가 진행 중이고 플레이어 차례여야 한다.
- 현재 라운드에서 아직 체인지를 사용하지 않았어야 한다.
- 플레이어 손에 비공개 카드가 정확히 1장 있어야 한다.
- 기존 비공개 카드를 제외한 덱·버린 더미에서 후보 2장을 확보할 수 있어야 한다.
- 다른 후보 선택 또는 라운드 판정을 처리 중이면 사용할 수 없다.

### 7.2 처리 순서

1. 체인지 시작 입력을 승인한다.
2. 기존 비공개 카드를 손에서 제거하되 후보 추첨이 끝날 때까지 별도 보관한다.
3. 일반 덱에서 후보 카드 2장을 뽑아 플레이어에게만 공개한다.
4. 전투 상태를 `PlayerChoosingChangeCard`로 바꾼다.
5. 플레이어가 후보 0 또는 1 중 하나를 선택한다.
6. 선택한 카드를 비공개 상태로 손에 넣는다.
7. 기존 비공개 카드와 선택하지 않은 후보를 버린 카드 더미로 이동한다.
8. 해당 라운드의 체인지 사용 여부를 완료로 표시한다.
9. 적 차례를 진행하고 이후 기존 전투 흐름으로 복귀한다.

후보를 표시한 시점부터 체인지는 취소할 수 없다. 후보 확인만 반복해 덱 정보를 얻는 행동을 막기 위한 결정이다.

### 7.3 후보와 정보 공개

- 두 후보의 숫자와 식별 정보는 플레이어 화면에만 보인다.
- 적 정책에는 후보, 선택 결과와 새 비공개 카드 숫자를 전달하지 않는다.
- 선택된 카드는 비공개 상태이므로 적 화면에는 `?`로 남는다.
- 기존 비공개 카드는 같은 체인지의 후보가 되지 않도록 후보 선택 완료까지 덱에 돌려놓지 않는다.
- 유효하지 않은 후보 번호 입력은 아무 카드도 이동하지 않고 선택 상태를 유지한다.

### 7.4 차례 소비와 제한 초기화

체인지 시작 입력은 후보 선택을 여는 중간 단계이며 별도의 차례를 소비하지 않는다. 유효한 후보 선택까지 완료되면 체인지 행동 하나가 끝난 것으로 보고 적 차례로 넘어간다.

라운드당 1회 제한은 새 라운드가 시작될 때 초기화한다. 전투 또는 런 재시작 시 이전 후보와 사용 여부를 남기지 않는다.

## 8. 화면과 입력

평상시 플레이어 차례에는 다음 네 행동을 표시한다.

- `HIT`
- `STAND`
- `FOLD (-1 SOUL)`
- `CHANGE (1/ROUND)`

체인지 후보 선택 중에는 일반 행동 버튼을 모두 숨기거나 비활성화하고 후보 2개만 선택할 수 있게 한다. 취소 버튼은 제공하지 않는다.

추가 표시 규칙은 다음과 같다.

- 영혼이 1일 때 폴드는 패배로 이어진다는 경고를 버튼에 표시한다.
- 체인지를 이미 사용했다면 해당 라운드에서 비활성화한다.
- 후보 카드 2장은 숫자가 보이도록 표시한다.
- 최근 라운드 결과에 폴드와 영혼 1 손실을 명시한다.
- 입력 처리 중에는 기존 입력 잠금 정책을 유지한다.

## 9. 기존 시스템과의 경계

| 계층 | 책임 |
| --- | --- |
| 전투 규칙 | 행동 승인, 카드 이동, 차례·라운드 상태, 피해와 승패 |
| 세션 | 폴드·체인지 입력을 현재 전투에 전달 |
| 표시 모델 | 행동 가능 여부, 후보와 최근 결과를 읽기 전용 값으로 변환 |
| View·Controller | 입력 전달, 버튼 활성화와 후보 표시 |
| 런·스테이지 진행 | 행동 전달 후 끝난 전투 결과와 플레이어 영혼을 기존 방식으로 동기화 |

런·스테이지 진행 계층은 폴드 피해나 체인지 카드 이동을 계산하지 않는다.

## 10. 완료 조건

- 폴드는 유효한 플레이어 차례에만 승인된다.
- 폴드는 영혼 1만 적용하고 라운드를 즉시 종료한다.
- 영혼 1에서 폴드하면 전투 및 진행 세션이 패배 상태가 된다.
- 체인지는 라운드당 정확히 한 번만 사용할 수 있다.
- 후보 2장 중 선택한 카드만 새 비공개 카드가 된다.
- 기존 비공개 카드와 미선택 후보는 버린 카드 더미로 이동한다.
- 후보는 플레이어에게만 보이고 적 공개 정보에는 노출되지 않는다.
- 유효하지 않은 입력은 카드·영혼·상태를 변경하지 않는다.
- 단독 전투와 런 전투에서 같은 행동이 작동한다.
- 기존 히트·스탠드, 승패, 재시작과 스테이지 진행 테스트가 계속 통과한다.
- 게임 관련 Unity Console Error·Exception이 없다.
